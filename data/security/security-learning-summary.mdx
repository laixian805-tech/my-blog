这段时间的网络安全学习，整体上经历了一个比较明显的过程：从最初的入门概念、靶场操作和漏洞复现，逐步走向对攻击原理、环境搭建、资产视角、内网安全和应急响应的综合理解。虽然现在距离真正的实战能力还有不小差距，但已经不再只是停留在“会跟着教程点点点”，而是开始慢慢意识到安全学习真正要补的是原理、代码、系统和场景感。

## 一、最初的入门阶段：从 DVWA、upload-labs 和 SQL 注入开始

前期最核心的学习内容，是围绕 DVWA、upload-labs 以及 SQL 注入等 Web 安全基础展开的。这个阶段的价值非常大，因为它让我第一次系统地接触到“漏洞为什么会出现”。

例如在学习 SQL 注入时，我逐渐理解了几个关键条件：用户输入是可控的、输入被带入数据库查询语句、后端没有做好过滤或过滤不严。再往后，像 `and 1=1` 与 `and 1=2` 的逻辑判断、`order by` 探测字段数、`union select` 联合查询、字符型注入与数字型注入的区别、盲注的回显特点，这些内容都不再只是零散技巧，而是开始形成一条比较清晰的攻击思路。

upload-labs 和文件包含相关内容则让我意识到，安全问题经常和开发细节直接相关。比如文件上传点为什么会被绕过、后端为什么需要严格校验、文件包含为什么可能导致敏感文件泄露甚至代码执行，这些都说明安全不是和开发分开的，它本质上就是对系统边界和输入处理的考验。

## 二、从“会利用”走向“想看懂代码”，开始意识到短板在哪里

在学习 DVWA 的过程中，我虽然能跟着完成很多关卡，但也越来越清楚地发现一个问题：如果只会复现，不会看 PHP 代码，那理解就会停留在表面。所以后面我专门补了一些 PHP 基础，包括表单处理、`$_GET` 与 `$_POST` 的差异、`include` 和 `require`、文件处理、过滤函数、JSON 处理等内容。

这部分学习虽然看起来偏开发，但对安全理解很有帮助。因为很多漏洞本来就不是凭空出现的，而是某个功能在实现时没有把边界处理好。比如 `$_SERVER["PHP_SELF"]` 的潜在风险、GET 方法对敏感参数的暴露、文件写入和包含的行为差异，这些内容都让我更能从“开发者写法”反推“攻击者利用点”。

也正因为如此，我后面越来越认同一个判断：如果想把 Web 安全继续往深处学，代码审计能力必须加强，不然很多学习都只会停留在工具和现象层。

## 三、靶场和漏洞复现，是把零散知识串起来的重要训练

除了 DVWA，我还打了不少 VulnHub 靶机，也部署过 Vulhub、远端 DVWA、NFS 挂载漏洞、Redis 未授权访问等环境。这个阶段最大的收获，不是单独掌握了某一个漏洞，而是开始意识到一条完整的渗透链条通常会涉及信息收集、服务识别、漏洞利用、提权、持久化、横向移动等多个环节。

例如在练习过程中，我接触到了 rbash 绕过、wpscan、命令执行、CSRF、盲注、文件包含、反弹 shell、netcat、curl、dig、DNS 隧道、socket 通信、socks 代理等知识点。它们单独看都只是一个个词，但放到靶场和攻击路径里理解时，就能慢慢形成“攻击面是怎么被一步步打开的”这种感觉。

另外，sqlmap 的使用也让我体会到自动化工具的价值，但也提醒了我不能过度依赖工具。因为工具能帮你跑出结果，但如果不知道背后的判断逻辑、参数意义和注入原理，一旦场景变化就容易失去方向。

## 四、学习视角开始从单点漏洞转向内网、资产与应急响应

后期安全学习的重心开始慢慢扩大。除了打靶和 Web 漏洞，我也开始看《内网安全攻防》、资产暴露面、FOFA、AI 资产识别、Linux 主机应急、完整性校验、持久化排查、日志与 IOC、挖矿排查等内容。这说明我的学习目标已经不只是“会打一台靶机”，而是开始尝试接近更真实的安全工作流。

这部分学习让我认识到，安全工作并不只是攻击。比如资产巡检要先考虑哪些 IP 更有价值、哪些暴露面更值得优先看；应急响应要关心的是日志、进程、文件、计划任务、异常网络连接、持久化方式；挖矿排查则需要从主机行为、外联地址、资源占用、可疑文件和进程关系去还原事件。这些内容比单一漏洞更贴近实际工作，因为它们要求同时具备系统视角、网络视角和证据链思维。

## 五、AI 安全相关内容，让我看到未来更想走的方向

在继续学安全的过程中，我也开始接触一些 AI 安全相关主题，比如提示词注入、AI 基础设施暴露面、基于大模型的蜜罐、自动化分析、智能体辅助安全等。这部分内容现在还在入门和尝试结合的阶段，但它给了我一个比较明确的方向感：与其单纯 all in 某一个传统安全细分，不如把 AI 与安全结合起来，形成自己的差异化能力。

比如蜜罐项目里的智能体设计、攻击链分析、AI 诱捕思路，以及对 AI 资产暴露面的关注，都让我觉得这个方向既贴近当前趋势，也和我前面的大模型项目学习可以形成联动。

## 六、这段网安学习里最真实的困难

第一，环境问题非常多。  
代理、端口、虚拟机、Docker、WSL、内网穿透、防火墙，这些问题经常会比漏洞原理本身更先把人卡住。

第二，代码能力仍然是明显短板。  
尤其是在 PHP、脚本分析、代码审计这块，如果不能把代码读顺，很多漏洞理解还是不够扎实。

第三，容易陷入“复现成功就以为学会了”。  
实际上只有能解释原理、看懂利用条件、知道防守思路，才算真正掌握。

## 七、下一步更值得继续深挖的方向

接下来如果要让网安学习继续往前走，我觉得重点应该放在下面几条线上：

1. 继续补 Web 开发与代码审计基础，尤其是 PHP、Python Web 和常见后端逻辑。
2. 把 SQL 注入、文件上传、文件包含、命令执行等常见漏洞做成自己的原理笔记。
3. 系统加强 Linux、网络协议、Docker、代理、内网环境这类基础设施能力。
4. 在打靶之外，多从资产梳理、应急响应、日志分析和真实业务暴露面角度学习。
5. 逐步把 AI 与安全结合，形成自己的长期主线。

整体来看，这一阶段的网安学习，已经让我从“知道一些漏洞名字”往“开始理解攻击与防守背后的系统逻辑”迈出了一步。后面真正要做的，不是盲目扩大学习面，而是把基础打扎实，把代码、环境和场景三件事慢慢补齐。
