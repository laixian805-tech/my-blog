这篇文章不是泛泛而谈的“建站教程”，而是我这次真正把个人网站从一个模板改造成自己能长期使用的知识博客时，按实际过程整理出来的完整记录。

我希望它能回答这几类问题：

- 为什么要做一个自己的个人网站
- 从零开始应该怎么选技术栈
- 一个个人博客到底要改哪些页面、哪些配置
- 我这次具体是怎么一步一步把它做出来的
- 部署上线以后，后面还能怎么优化、怎么更稳定、怎么更好维护

如果以后我自己回头看，这篇文章也能作为一份项目复盘和运维手册。

## 一、我为什么要自己做一个个人网站

我想做这个网站，不只是为了“有个博客”。

更准确地说，我想做的是一个属于自己的中文知识站，用来长期沉淀这些内容：

- 后端学习笔记
- 数据库学习笔记
- 算法刷题过程
- 全栈项目理解
- 网络安全与靶场练习记录
- 以后可能继续扩展的项目总结和工具清单

如果这些内容只散落在本地笔记软件、零散 Markdown、临时文档里，最大的几个问题是：

- 很难长期整理成体系
- 很难形成公开作品
- 很难回头复用
- 很难让别人真正看到我的学习路径

所以我希望这个网站具备三个核心目标：

1. 自己能持续写
2. 别人能稳定访问
3. 结构清楚，后面愿意继续维护

## 二、这次建站的总体思路

我没有选择完全从零写一个博客系统，而是选择在成熟模板基础上进行深度改造。

原因很简单：

- 从零写博客框架，前期会花很多时间在基础设施上
- 我当前更需要的是“尽快有一个能用、能写、能部署、能继续迭代”的站
- 模板可以帮我省掉很多底层工作，例如 MDX、SEO、RSS、文章路由、分页等

所以这次我的思路是：

1. 先找一个成熟、适合技术博客的 Next.js 模板
2. 先把内容发布链路跑通
3. 再把页面结构改成更贴近我自己使用方式的样子
4. 最后再解决部署、统计、可维护性和稳定性问题

## 三、我最后采用的技术栈

当前站点的核心技术栈是：

- `Next.js 15`
- `React 19`
- `Tailwind CSS 4`
- `Contentlayer 2`
- `MDX`
- `Pliny UI`
- `GitHub Pages`
- `Busuanzi`

可以把它理解成下面这套分工：

- `Next.js`：负责整个网站框架、路由、构建和静态导出
- `React`：负责页面组件
- `Tailwind CSS`：负责样式组织
- `MDX`：负责把文章当作内容源来写
- `Contentlayer`：负责把 `data/blog/*.mdx`、`data/security/*.mdx` 这些内容编译成站点可读取的数据
- `Pliny`：提供博客站常见的基础能力
- `GitHub Pages`：负责静态托管
- `Busuanzi`：负责前端 UV/PV 展示

## 四、为什么最后是这套组合

### 1. 为什么用 Next.js

因为它在内容站、文档站、博客站这类项目上成熟度很高。

它能给我的几个实际好处是：

- 路由能力成熟
- 页面组件组织方便
- SEO 支持不错
- 静态导出能力可用
- 周边生态很完整

### 2. 为什么用 MDX

我写博客本质上还是想保留 Markdown 体验。

MDX 的好处在于：

- 普通文章可以像 Markdown 一样写
- 以后如果我想在文章里插自定义组件，也有扩展空间
- 内容和代码可以放在同一个仓库里，版本管理很顺手

### 3. 为什么用 Contentlayer

因为我不想手写大量“读文件、解析 frontmatter、生成列表”的基础逻辑。

`Contentlayer` 会把内容目录编译成结构化数据，让页面可以直接取用文章标题、摘要、slug、日期等字段。

### 4. 为什么最后选择 GitHub Pages

我一开始也考虑过继续用 `Vercel`，因为它对 Next.js 支持更顺手。

但这次最后切到 `GitHub Pages`，主要是因为：

- 对纯内容型博客来说，静态导出已经够用
- GitHub 仓库和 Pages 可以形成很直接的发布链路
- 对个人项目来说，维护成本低
- 对我这种以写文章为主、不是以复杂服务端逻辑为主的站点来说足够实用

### 5. 为什么统计用 Busuanzi

我这次明确不想再单独维护一套后端统计服务。

如果用自建统计方案，通常要额外处理：

- 数据采集
- 存储
- 防刷
- 鉴权
- 部署
- 成本

对当前阶段来说，这些不是我最需要投入精力的地方。

所以我最后用了更轻量的方案：

- 首页首屏先显示兜底值
- 浏览器端加载 `Busuanzi` 脚本
- 如果脚本可用，就覆盖 UV/PV
- 如果脚本临时不可用，页面也不会空白

这是一种更偏“够用优先、维护成本低”的取舍。

## 五、我这次具体做了哪些改造

这部分是最接近真实项目过程的内容。

### 1. 先确定站点定位

我没有把它做成一个大而全的网站，而是明确成一个知识型博客：

- 首页展示个人信息、内容统计、最新文章
- `Blog` 放普通技术文章
- `Security` 放网安笔记
- `Nav` 放常用网站导航
- `Projects` 放项目入口和长期可展示内容

这个定位非常重要。

因为一旦定位不清，后面的页面结构、内容分类、导航设计都会反复摇摆。

### 2. 把内容拆成两类：Blog 和 Security

这是我这次建站里非常关键的一步。

我不想把所有内容都混在一个列表里，所以我把内容源拆成了两套：

- `data/blog/`：普通博客文章
- `data/security/`：网络安全学习笔记

这样做的好处是：

- 首页最新文章不会被安全笔记刷屏
- `/blog` 和 `/security` 各自职责清楚
- 搜索时也可以按内容类型理解结果
- 后续如果继续扩内容，不容易混乱

### 3. 改顶部导航

顶部导航我最后保留了这几个入口：

- `Home`
- `Blog`
- `Nav`
- `Projects`
- `Security`

这个导航结构的含义是：

- `Blog`：输出学习型文章
- `Nav`：沉淀常用资源入口
- `Projects`：展示有成果感、可对外看的项目
- `Security`：独立承载网安学习内容

### 4. 改首页

首页是整个网站里最像“门面”的地方，所以我在这里做了相对多的改造。

当前首页主要包含这些块：

- 头像和 GitHub 按钮
- 一段简短定位文案
- 站点统计区
- 最新文章列表

其中统计区又分成两部分：

- 内容统计：总文章数、博客文章数、网安笔记数、建站天数
- 站点状态：独立访客、总访问量、最后更新

这样首页既能体现“我是做什么的”，也能体现“这个站点是活的、在持续更新”。

### 5. 给文章页补目录能力

文章越写越长以后，目录就很重要。

所以我给文章详情页加上了目录能力：

- 桌面端右侧吸顶
- 移动端折叠展开

这个改动虽然不是最显眼的，但对阅读体验提升很大。

### 6. 做全站搜索

如果只是写几篇文章，搜索不一定刚需。

但只要内容开始累积，搜索就会变成站点体验里很重要的一部分。

我这次把搜索入口统一放到了：

- 顶栏桌面端
- 移动端菜单顶部

这样用户无论在哪个页面，都能快速找内容。

### 7. 改 Projects 页

我没有把 `Projects` 页做成“纯装饰”页面，而是把它当作一个项目成果入口。

当前它更像一组项目卡片：

- 可以链接到首页
- 可以链接到代表性的技术文章
- 也可以继续补充真正的项目总结和专题文章

这次你让我补的这篇“自建个人网站全过程”，我也会把它挂进 `Projects`，让这个页面更像一个作品入口页。

## 六、这个网站现在的目录结构是怎么理解的

我现在是这样理解当前项目结构的：

```text
app/                 页面路由
components/          页面组件
data/blog/           普通博客文章
data/security/       网安笔记
data/projectsData.ts Projects 页卡片数据
data/siteMetadata.js 站点基础配置
layouts/             文章布局
public/static/       图片、favicon 等静态资源
```

这套结构最大的优点是职责分离清楚：

- 页面归页面
- 组件归组件
- 内容归内容
- 配置归配置

后面继续写的时候，定位文件会很快。

## 七、技术上它是怎么工作的

这部分是“实现原理”的简化版说明。

### 1. 内容是怎么变成页面的

我平时写的文章，本质上就是放在：

- `data/blog/*.mdx`
- `data/security/*.mdx`

构建时，`Contentlayer` 会读取这些文件，把 frontmatter 和正文转成结构化内容数据。

然后：

- 首页读取最新的普通博客文章
- `/blog` 读取博客内容列表
- `/security` 读取安全笔记列表
- 详情页按 slug 渲染对应文章

所以整个网站虽然看起来像“动态博客”，但实际是“构建时生成好的静态页面”。

### 2. 首页统计是怎么实现的

首页内容统计是本地计算出来的，例如：

- 总文章数
- Blog 数量
- Security 数量
- 建站天数

站点状态则分成三项：

- `独立访客 (UV)`
- `总访问量 (PV)`
- `最后更新`

其中：

- `最后更新` 来自 `NEXT_PUBLIC_LAST_UPDATED`
- `UV/PV` 首屏显示 `siteMetadata.js` 里的兜底值
- 页面加载后再由 `Busuanzi` 脚本覆盖显示值

### 3. GitHub Pages 是怎么部署的

当前仓库已经配置了 GitHub Actions 工作流。

发布流程大致是：

1. 代码推送到 `main`
2. GitHub Actions 自动执行构建
3. 构建时启用静态导出模式
4. 生成 `out/` 目录
5. GitHub Pages 发布 `out/` 内容

核心环境变量包括：

```bash
EXPORT=1
BASE_PATH=/my-blog
UNOPTIMIZED=1
SITE_URL=https://laixian805-tech.github.io/my-blog
NEXT_PUBLIC_LAST_UPDATED=<git log -1 --format=%cs>
```

这几项很关键：

- `EXPORT=1`：告诉 Next.js 走静态导出
- `BASE_PATH=/my-blog`：因为 GitHub Pages 项目页不是根域名，而是子路径
- `UNOPTIMIZED=1`：静态导出时避免图片优化服务依赖
- `SITE_URL`：统一生成正确的 SEO 绝对地址
- `NEXT_PUBLIC_LAST_UPDATED`：把最近一次提交日期显示到首页

## 八、如果我以后要继续更新网站，具体该怎么操作

这里我把日常使用流程也写清楚。

### 1. 本地开发

在项目根目录：

```bash
yarn
yarn dev
```

然后打开：

```text
http://localhost:3000
```

### 2. 新增一篇文章

如果是普通技术文章，就放到：

```text
data/blog/
```

如果是网安笔记，就放到：

```text
data/security/
```

frontmatter 至少要有：

```md
---
title: '文章标题'
date: '2026-03-18'
tags: ['标签1', '标签2']
draft: false
summary: '一句摘要'
authors: ['default']
layout: PostLayout
---
```

### 3. 本地检查

写完以后建议至少跑一次：

```bash
yarn build
```

如果准备验证 GitHub Pages 结果，就再跑一次：

```powershell
$env:EXPORT='1'
$env:BASE_PATH='/my-blog'
$env:UNOPTIMIZED='1'
$env:SITE_URL='https://laixian805-tech.github.io/my-blog'
$env:NEXT_PUBLIC_LAST_UPDATED='2026-03-18'
yarn build
```

### 4. 提交并推送

提交代码的基本流程是：

```bash
git status
git add .
git commit -m "feat: update homepage stats and add website-building guide"
git push origin main
```

### 5. GitHub Pages 自动发布

只要仓库 `Settings > Pages` 已经设置成 `GitHub Actions`，推送到 `main` 后一般就会自动发布。

可以去这里看构建状态：

- `GitHub 仓库 > Actions`

发布成功后访问：

```text
https://laixian805-tech.github.io/my-blog
```

## 九、这个网站现在还可以怎么继续优化

一个个人网站能上线，只是第一步。

后面真正拉开差距的，是持续优化。

### 1. 内容层优化

可以继续做的内容优化包括：

- 给系列文章建立更强的专题化结构
- 增加“相关文章推荐”
- 给 Security 笔记增加更清楚的难度和分类
- 给 Projects 增加真正的项目复盘文档
- 给首页增加“近期重点更新”

### 2. 体验层优化

可以继续做的体验优化包括：

- 进一步打磨移动端排版
- 给目录增加阅读进度高亮
- 给搜索结果页增加更清楚的分类提示
- 给文章卡片增加阅读时间
- 给代码块增加复制按钮和高亮主题优化

### 3. SEO 优化

目前站点已经有基础 SEO，但后面还能继续做：

- 更细化的页面描述
- 文章级别的社交分享图
- 更系统的关键词布局
- 更规范的内部链接
- 更丰富的结构化数据

### 4. 统计与运营优化

如果以后我对统计精度要求更高，可以考虑从 `Busuanzi` 逐步升级到更完整的方案，例如：

- Google Analytics
- Umami
- Plausible
- 自建统计服务

当前阶段 `Busuanzi` 的优势是轻量；后期如果更关注长期趋势和来源分析，再升级更合适。

### 5. 稳定性优化

这部分其实非常重要。

如果我希望网站未来更稳定，最值得优先考虑的是：

1. 绑定自定义域名  
   这样访问地址更稳定，也更像真正的个人品牌站。

2. 在自定义域名前面接一层 Cloudflare  
   可以获得更灵活的 DNS、缓存、安全规则和访问优化能力。

3. 保持纯静态优先  
   动态能力越少，故障面通常越小；对个人博客来说，静态导出是非常稳的路线。

4. 保持内容和代码同仓库版本管理  
   出问题时更容易回滚，也更方便审计和备份。

5. 给 GitHub Actions 保持最小但明确的发布流程  
   流程越简单，越不容易出现“某个隐藏服务挂了整站发不出去”的情况。

### 6. 维护性优化

维护性经常被低估，但它会决定我愿不愿意继续写下去。

我后面最想继续优化的是：

- 把常改配置收敛到少数文件里
- 给关键页面组件做好职责拆分
- 把常用操作写进 README
- 保持文章命名、frontmatter、目录结构一致
- 尽量避免为了炫技引入过多复杂依赖

## 十、如果我以后想让它更“像一个长期项目”，可以往哪几个方向升级

如果不是只做一个简单博客，而是想把它做成真正长期维护的个人站，我觉得最值得投入的升级方向是下面这些。

### 1. 自定义域名

这是优先级很高的一步。

原因包括：

- 访问地址更专业
- 更利于长期品牌沉淀
- 后面切换托管平台时更平滑

### 2. 评论系统

如果以后希望有互动，可以考虑：

- `Giscus`
- `Gitalk`
- 其他第三方评论服务

但评论系统也意味着更多噪音和维护成本，所以我现在没有优先上。

### 3. 监控和可用性检查

后面如果站点访问量增加，可以补这些能力：

- 可用性监控
- GitHub Actions 发布失败提醒
- 页面打开速度监控
- 外链和资源失效检测

### 4. 图片与资源策略

当前已经能正常运行，但以后还可以继续优化：

- 压缩图片体积
- 统一静态资源命名
- 规范 favicon、分享图和头像素材
- 如果资源变多，再考虑独立静态资源托管

### 5. 站内内容资产化

这是长期价值最大的一项。

我希望它最后不只是“有几篇文章”，而是逐渐形成：

- 学习路径
- 专题专栏
- 项目复盘
- 导航资源沉淀
- 安全学习资料库

当内容资产逐渐成体系，网站价值就会明显高于零散笔记。

## 十一、我这次建站最大的收获

这次最大的收获不是“我上线了一个网站”，而是我把下面这些事情真正打通了：

- 我知道一个个人知识站应该怎么定位
- 我知道内容站的技术栈为什么这么选
- 我知道博客、导航、项目、安全笔记这些模块怎么拆
- 我知道静态导出和 GitHub Pages 这条链路是怎么走的
- 我知道一个看起来不大的个人站，真正要长期维护时应该关心什么

更重要的是，它现在不是一个摆设，而是一个我可以继续写、继续迭代、继续展示成长过程的站点。

## 十二、我给未来自己的行动清单

为了让这个站点后面真的越做越好，我给自己列一个可执行清单：

1. 持续更新高质量文章，不让首页长期停更
2. 每写完一篇文章就跑一次构建，避免线上才发现问题
3. 继续补充 `Projects` 页，把真正有代表性的内容挂出来
4. 后续优先考虑自定义域名和访问稳定性优化
5. 当内容量足够以后，再升级更精细的统计和监控方案

如果把这个网站看作一个长期项目，那么它最重要的不是“一次性做完”，而是每次都往前走一小步，并且每一步都让结构更稳、内容更清楚、维护更轻松。
