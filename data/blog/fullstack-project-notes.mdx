这篇文章更像一份“看项目之前的地图”。我不是想先背代码，而是先把一个全栈项目里最核心的角色和数据流转顺序看清楚。

## 先用一句人话说清楚：全栈项目是什么

一个教学型全栈项目，本质上是把前端和后端放在同一个仓库里，让你能看到从页面交互到数据返回的完整链路。

我现在更愿意把它理解成一个闭环系统：

- 前端负责“用户看见什么、点了以后怎么变化”
- 后端负责“请求进来以后由谁处理、返回什么结果”
- 数据层负责“保存和读取数据”
- 共享类型负责“让前后端对数据格式有一致理解”

这类项目适合入门的原因不是功能多，而是 **链路完整**。从打开页面，到发送请求，到后端处理，再到页面更新，这一整圈是闭合的。

## 先记住四个角色

### 1. 浏览器

浏览器不只是“显示网页”的软件，它更像一个会执行前端代码的运行环境。它会：

- 接收服务器发来的 HTML、CSS、JavaScript
- 把 HTML 解释成页面结构，把 CSS 解释成样式
- 执行 JavaScript，让页面根据点击、输入、跳转产生动态变化

如果没有 JavaScript，很多现代页面就只是一张不会动的海报；有了 JavaScript，页面才真正变成一个交互界面。

### 2. 前端

以 React 为代表的前端框架，最重要的价值是帮助我们用组件组织界面。

我现在更习惯这样理解它：

- 不是手工去改页面上的每个节点
- 而是描述“当前数据是什么，界面应该长什么样”

比如用户列表：

- 数据为空，就显示“暂无用户”
- 数据有 3 条，就渲染 3 行
- 新增一条数据，界面会跟着更新

### 3. 后端

后端的核心职责，是持续等待请求并决定怎么回应。以 `Express` 这类框架举例，它通常负责：

- 判断请求是要页面还是要数据
- 决定交给哪个路由处理
- 最后把页面、JSON 或错误信息返回回去

所以第一层理解其实很朴素：**服务器就是一段一直在等待请求、收到请求后再决定怎么回应的程序。**

### 4. 共享类型

`TypeScript` 在全栈项目里的真正价值，不是“它很高级”，而是它能提前说清楚：

- 数据应该长什么样
- 哪些字段是必须的
- 字段类型是什么

这能减少前后端对同一份数据理解不一致的问题。

## 一个按钮被点击后，链路是怎么走的

如果把一次最普通的交互拆开，大致可以看成下面这条链：

1. 用户在浏览器里点击按钮
2. 前端组件捕获事件
3. 前端组织请求数据并发给后端
4. 后端路由接收请求
5. 后端执行业务逻辑，读取或修改数据
6. 后端把结果返回给前端
7. 前端收到结果后更新页面状态
8. 浏览器重新展示更新后的界面

当我能把这条链用自己的话复述出来时，看全栈项目就不会再只盯着某个单独文件。

## 为什么先看角色，比先点代码更重要

很多人一上来就想把整个仓库从头到尾读完，但这往往会很快迷路。

对我来说，更有效的顺序是：

1. 先认清项目里的几个核心角色
2. 再搞清楚请求是怎么流动的
3. 最后再回头看每一层的具体实现

这样做的好处是，不会把“浏览器的职责”“前端组件的职责”“后端接口的职责”混在一起。

## 我现在给自己的观察重点

以后再看一个新全栈项目时，我会先回答这些问题：

- 浏览器这边主要负责什么
- 前端状态是在哪里保存和更新的
- 后端入口在哪里
- 数据是怎么被读取和返回的
- 前后端用什么方式约束数据格式

只要这几个问题先有答案，项目就已经不再是“密密麻麻的一堆文件”，而是一条可以顺着走下去的完整路径。
