# my-blog 项目笔记：一个中文知识博客的搭建、部署与长期维护记录

这份笔记不是博客文章，也不是正式项目书，而是我后面自己回看、补内容、做维护、写项目总结时都能直接翻的版本。  
写法尽量按“先讲人话，再讲结构，再讲技术，最后讲注意事项和坑点”的顺序来，不追求论文味，追求后面一眼能看懂、出了问题能快速对照。

我希望这篇笔记至少能回答 5 个问题：

1. 这个项目到底是什么
2. 它现在是怎么工作的
3. 它用了哪些技术，各自负责什么
4. 它相对普通博客模板做了哪些改造
5. 部署、维护、写文档时最容易踩哪些坑

---

## 1. 先用一句人话说清楚：这个项目是什么

`my-blog` 本质上是一个**面向长期积累的中文个人知识博客项目**。

它不是那种只放几篇文章的静态主页，也不是只追求“先搭起来再说”的一次性练手站。  
它真正想做的是把下面这些内容长期收进同一个站点里：

- 普通技术文章
- 项目复盘与项目入口
- 常用网站导航门户
- 网络安全学习笔记
- 后续还能继续扩展的个人公开内容

所以更准确地说，它不是“一个博客模板”，而是一个**以博客为核心、同时承载内容沉淀、项目展示和持续维护的个人知识站**。

---

## 2. 这个项目到底在做什么

如果只看表面，这个项目像是一个 Next.js 博客；  
如果看深一点，它其实在做的是下面这件事：

**把零散的学习记录、项目总结、网安笔记和常用资源入口，整理成一个可以持续写、持续部署、持续维护的个人站点。**

这个目标可以拆成 4 层理解：

### 2.1 第一层：把内容长期沉淀下来

项目现在不只是为了“发文章”，而是为了把平时积累的内容逐渐沉淀成公开资产，比如：

- 后端学习笔记
- 全栈项目理解
- 建站复盘
- GitHub Pages 相关排障手册
- 网安靶场和漏洞练习记录

如果这些内容只散落在本地 Markdown、临时文档和笔记软件里，后面会越来越难复用，也越来越难展示。

### 2.2 第二层：把不同类型的内容分流清楚

这个项目没有把所有东西混进一个列表里，而是明确拆成了几块：

- `Blog`：普通技术文章
- `Security`：网络安全笔记
- `Nav`：常用网站导航门户
- `Projects`：项目入口和成果展示

也就是说，这个站点不是“一个列表塞所有内容”，而是从一开始就在做内容分层。

### 2.3 第三层：把内容发布链路跑通

这个项目的内容不是手工往 HTML 里硬写，而是走一条比较清楚的链路：

1. 用 MDX 写内容
2. 通过 Contentlayer 把内容编译成结构化数据
3. 由 Next.js 路由和页面组件渲染成站点页面
4. 最后导出为静态站点并发布到 GitHub Pages

这样做的好处是，后面继续写内容时不用再碰底层页面结构，主要维护内容和少量配置就行。

### 2.4 第四层：把它做成一个可维护项目

这个项目一个很重要的目标不是“能上线一次”，而是“后面还能继续稳定维护”。

所以它后面逐渐补了这些东西：

- README 形式的维护文档
- 更新记录文档
- GitHub Pages 排障清单
- 自定义域名迁移手册
- Cloudflare 后续优化规划

也就是说，这个项目已经不是单纯写页面，而是在把部署、维护和文档沉淀也当成项目本体的一部分。

## 3. 项目整体结构与工作方式

看这个项目时，最容易乱的地方是：页面、内容、配置和静态资源都分散在不同目录里。  
所以更适合按“它们在项目里分别负责什么”来看，而不是只背目录名。

### 3.1 页面路由层：`app/`

`app/` 这一层主要负责站点的页面入口和路由组织，比如：

- 首页
- `/blog`
- `/security`
- `/nav`
- `/projects`
- `/search`

可以把它理解成“这个站有哪些页面、每个页面从哪里进”的总入口。

### 3.2 组件层：`components/`

`components/` 负责承载各个页面会复用的界面能力，比如：

- 顶部导航
- 搜索表单
- 文章目录
- 首页状态区
- 导航门户卡片区

如果 `app/` 决定“页面是什么”，那 `components/` 决定“页面具体怎么拼起来”。

### 3.3 内容层：`data/blog/` 和 `data/security/`

这两个目录是这个项目最关键的内容源：

- `data/blog/`：普通博客文章
- `data/security/`：网络安全笔记

它们表面看只是两个文件夹，但项目语义完全不同。  
因为一篇内容放在哪个目录里，直接决定它会出现在：

- 首页还是不会出现在首页
- `/blog` 还是 `/security`
- 搜索结果里的 `Blog` 分组还是 `Security` 分组

### 3.4 项目与导航数据层：`data/projectsData.ts` 和 `data/portalData.ts`

这个项目有两个很有意思的数据入口：

- `data/projectsData.ts`：控制 `/projects` 页里展示哪些项目卡片
- `data/portalData.ts`：控制 `/nav` 页面里的分类和站点卡片

这说明站点不只是“写文章”，还在把项目展示和资源导航也做成结构化入口。

### 3.5 配置层：`data/siteMetadata.js`

这个文件可以理解成整个站点的基础配置中心，里面统一维护：

- 站点标题
- 描述
- 仓库地址
- 正式站点地址
- 社交链接
- 首页状态区的默认值
- 最后更新时间相关配置

后面很多看起来像“页面显示问题”的东西，根源其实都在这一层配置。

### 3.6 静态资源层：`public/static/`

这一层主要放：

- 头像
- 分享图
- 站点图片
- 导航门户图标

它的作用不是“随便放点图片”，而是把所有需要随站点一起发布的静态资源统一收口。

### 3.7 内容是怎么变成页面的

这个项目最值得先记住的一条主链路是：

1. 我平时写的内容，本质上是 `data/blog/*.mdx` 和 `data/security/*.mdx`
2. Contentlayer 构建时会读取这些文件，提取 frontmatter、正文和目录结构
3. Next.js 页面读取编译后的内容数据，渲染成列表页和详情页
4. 构建完成后导出成静态页面
5. GitHub Pages 负责托管最终导出的站点内容

所以它虽然看起来像“动态博客”，实际核心路线还是**内容驱动的静态站点**。

## 4. 这个项目用了哪些技术，它们各自负责什么

这部分不要只背技术名，关键是知道“它们在这个项目里具体干什么”。

### 4.1 `Next.js 15`

`Next.js` 是整个网站的框架骨架，主要负责：

- 页面路由
- 构建流程
- 页面渲染
- 静态导出
- SEO 相关能力

如果把这个项目比作一个站点系统，`Next.js` 就是最底层的框架骨架。

### 4.2 `React 19`

`React` 负责页面组件的组织和渲染。

这个项目里很多实际可见的界面能力，本质上都是 React 组件在协作，比如：

- 顶栏
- 搜索框
- 首页状态区
- 导航门户卡片
- 文章目录

### 4.3 `Tailwind CSS 4`

`Tailwind CSS` 主要负责样式组织。

它在这个项目里的意义，不只是“写样式更快”，而是让页面结构、响应式布局和细节调整更容易跟组件放在一起维护。

### 4.4 `MDX`

`MDX` 是内容编写的核心方式。

它的价值在于：

- 平时可以按 Markdown 习惯写文章
- 后面如果需要，也能在内容里接入组件能力
- 内容和代码都放在同一个仓库里，版本管理很顺

对这个项目来说，`MDX` 不是装饰，而是内容生产方式本身。

### 4.5 `Contentlayer 2`

`Contentlayer` 负责把内容文件编译成结构化数据。

也就是说，项目不用手写一堆“扫描文件、读 frontmatter、生成 slug、拼文章列表”的底层逻辑，而是把这些事情交给它来做。

### 4.6 `Pliny UI`

这个项目最开始是基于技术博客模板改造出来的，`Pliny UI` 提供了很多博客站常见的基础能力，例如：

- 文章布局
- 内容读取辅助
- 目录和列表相关能力

它更像是博客模板提供的基础设施，而不是项目后期的全部逻辑。

### 4.7 `Yarn 3`

`Yarn` 负责依赖安装和脚本管理。

这个项目常用的命令基本都围绕它：

- `yarn`
- `yarn dev`
- `yarn build`
- `yarn serve`
- `yarn test`

### 4.8 `GitHub Actions + GitHub Pages`

这两者一起构成当前项目的自动部署链路：

- `GitHub Actions`：构建并输出静态产物
- `GitHub Pages`：托管最终静态站点

对这个博客来说，这条路线的重点不是“最炫”，而是“够用、稳定、维护成本低”。

### 4.9 `Busuanzi`

`Busuanzi` 负责首页前端 UV/PV 展示。

它在这个项目里的定位不是精细化统计平台，而是一个轻量、够用的前端访问量展示方案。  
为了避免第三方脚本不稳定时页面空掉，项目里还保留了默认兜底值。

## 5. 这个项目相对普通博客模板做了哪些改造和创新点

这部分我不按“原创算法”那种口气写，而按“相对原模板做了哪些有意识的结构改造”来记。

### 5.1 改造点一：把默认英文技术博客改成中文知识站

这不是简单换个标题和颜色，而是把站点整体定位改了：

- 内容语言转成中文
- 内容目标从“技术博客模板演示”变成“长期知识沉淀”
- 页面入口围绕自己的学习路径重构

这一改造决定了后面所有页面结构和内容组织方式。

### 5.2 改造点二：用 `Security` 替代传统 `About`

很多博客模板会有一个比较静态的 `About` 页面，信息量有限。  
这个项目把它改成了 `Security` 主入口，用来单独承载网络安全学习笔记。

这样做的价值在于：

- 网安内容不会和普通博客混成一团
- 站点结构更贴近自己的真实学习路线
- `/security` 这一块可以长期独立扩展

### 5.3 改造点三：用 `Nav` 替代标签体系

原模板常见的是 `Tags` 体系，这个项目把它换成了 `Nav` 导航门户页。

也就是说，它没有优先做文章标签站内浏览，而是优先做了一个常用网站导航入口。  
这体现的是一种很实际的取舍：

- 当前更需要常用资源入口
- 标签体系不是首版刚需
- 导航页更符合自己日常使用习惯

### 5.4 改造点四：把搜索统一到全站顶栏

这个项目没有让不同页面各自带一套搜索，而是把站内搜索统一收口到：

- 桌面端顶栏
- 移动端菜单顶部

这样做的价值是：

- 搜索入口更稳定
- 页面职责更清楚
- 用户不需要记“某个页面才有搜索”

### 5.5 改造点五：把内容拆成两条独立内容链路

`blog` 和 `security` 的分流，是这个项目非常关键的设计点。

它带来的实际效果包括：

- 首页文章流不会被安全笔记刷屏
- 普通文章和安全笔记各有自己的列表和详情路由
- 搜索结果也能按内容类型理解

这个改造看起来不花哨，但对长期维护特别重要。

### 5.6 改造点六：首页状态区不只展示文章数

首页状态区不是只列内容数量，而是把几种信息揉在一起：

- 内容统计
- UV/PV
- 最后更新时间

这让首页不只是“一个头像加文章列表”，而更像一个持续更新中的项目主页。

### 5.7 改造点七：为 GitHub Pages 项目页场景做了专项适配

这个项目不是停在本地能跑，而是专门为 GitHub Pages 项目页做了很多适配，例如：

- 静态导出
- `basePath`
- 搜索页兼容
- 图片路径和链接路径修正

也就是说，它不是“用默认配置碰碰运气上线”，而是在认真适配子路径静态站点场景。

### 5.8 改造点八：把运维知识也沉淀成项目内容

这个项目有一个很实用的特点：  
不是只改代码，还把后续最容易忘的运维知识写成了站内文章或仓库文档，比如：

- GitHub Pages 排障清单
- 自定义域名迁移手册
- Cloudflare 后续优化规划
- README 维护说明
- 更新记录

这说明它已经开始从“建站”走向“长期维护”视角了。

## 6. 这个项目现在是怎么部署和维护的

这一节很重要，因为这个项目真正稳定，靠的不只是页面代码，还靠整条发布和维护链路。

### 6.1 本地开发怎么跑

最基本的本地开发流程是：

```bash
yarn
yarn dev
```

然后默认访问：

```text
http://localhost:3000
```

这一步的意义不是“看页面有没有打开”这么简单，而是先确认：

- 依赖是否完整
- 本地开发脚本是否可用
- 主要页面和内容读取链路是否正常

### 6.2 本地构建怎么检查

每次发文、改页面、改配置以后，最该先跑的命令是：

```bash
yarn build
```

因为这个项目很多问题不是开发环境立刻暴露，而是在构建阶段才会报出来，比如：

- frontmatter 写错
- 路由路径处理有问题
- 静态导出兼容性问题
- 内容引用或组件导入错误

### 6.3 GitHub Pages 的发布链路

当前项目的重点发布路线是：

- 写内容或改代码
- 推送到 GitHub
- GitHub Actions 自动构建
- 生成静态产物
- 发布到 GitHub Pages

这条链路里最关键的环境变量包括：

```text
EXPORT=1
BASE_PATH=/my-blog
UNOPTIMIZED=1
SITE_URL=<正式站点地址>
NEXT_PUBLIC_LAST_UPDATED=<最近一次提交日期>
```

它们各自负责的事情可以理解成：

- `EXPORT=1`：走静态导出模式
- `BASE_PATH=/my-blog`：适配 GitHub Pages 项目页子路径
- `UNOPTIMIZED=1`：避免静态导出依赖图片优化服务
- `SITE_URL`：生成正确的绝对地址和 SEO 信息
- `NEXT_PUBLIC_LAST_UPDATED`：把最后更新时间带到前端展示

### 6.4 为什么这个站点是“静态导出优先”

因为当前站点核心是内容展示，而不是复杂后端交互系统。  
所以对这个项目来说，静态导出路线有几个非常实际的优势：

- 托管简单
- 成本低
- 结构稳
- 故障面小
- 更适合内容型网站

这也是它当前重点围绕 GitHub Pages 做适配的原因。

### 6.5 Vercel 在这个项目里的位置

这个项目并不是完全否定 Vercel。

仓库文档里也明确保留了面向 Vercel 的思路，因为：

- 它对 Next.js 支持更省心
- 某些场景下部署体验更顺

但就当前项目状态来说，重点已经放在 **GitHub Pages 场景下的静态导出和可维护性**，而不是把 Vercel 写成唯一正式路线。

## 7. 注意事项与常见踩坑总结

这一节是后面最值得反复看的部分。  
很多问题不是“不会写代码”，而是“项目真的跑起来、写久了、维护久了以后，很容易在这些地方出事”。

---

### 7.1 开发环境与编码类注意事项

#### 1. `yarn dev` 在 Windows PowerShell 下要先确认能直接跑

这个项目之前遇到过一个很典型的问题：

- `yarn dev` 报错 `Unbound variable "PWD"`

当前脚本已经按仓库状态处理过，但这不代表所有环境都百分之百不会再碰到。  
尤其是换终端、换壳层、换环境以后，第一件事还是要先验证：

```bash
yarn dev
```

能不能在当前环境直接跑通。

#### 2. Markdown / MDX 文件一定要统一保存为 UTF-8

这个项目内容以中文为主，所以文件编码非常关键。  
如果编码不一致，最常见的问题就是：

- 中文乱码
- 局部字符异常
- 内容明明写了，但构建和预览结果不对

尤其是这些目录更要注意：

- `data/blog/*.mdx`
- `data/security/*.mdx`
- 其他本地项目笔记 Markdown

#### 3. 外部文档导入时不要保留本地绝对图片路径

这一条是非常容易踩坑的。

如果原文里还留着这种路径：

```text
C:\Users\...\Typora\...
```

那本地也许能看，部署后一定会失效。  
更稳的做法是：

- 把图片复制到站点资源目录
- 改成站内相对路径
- 或者干脆删掉无必要图片引用

#### 4. 文章文件名尽量用 ASCII slug

标题当然可以写中文，但文件名尽量不要直接用中文。  
原因很实际：

- URL 编码更容易变复杂
- 静态导出后路径更容易出歧义
- 复制链接和排查问题时更麻烦

所以更稳的习惯是：

- 中文标题
- 英文或 ASCII 文件名

---

### 7.2 内容组织与构建类注意事项

#### 1. `data/blog/` 和 `data/security/` 是两套独立内容源

这个项目最容易被误操作的一点，就是把内容放错目录。

如果普通文章误放进 `data/security/`，或者安全笔记误放进 `data/blog/`，后果不是“小问题”，而是会直接影响：

- 展示入口
- 首页是否出现
- 列表页归属
- 搜索结果分组

所以新增内容时，第一件事不是写标题，而是先确认它属于哪一类内容。

#### 2. frontmatter 里的 `authors` 和 `layout` 不能乱写

这个项目的内容并不是“正文写了就行”，frontmatter 写错也可能直接把构建打断。  
尤其是：

- `authors`
- `layout`

如果作者 slug 不存在，或者布局名写错，构建阶段就可能直接报错。

#### 3. 每次改完内容或配置，先跑一次 `yarn build`

这是这类内容站最值得养成的习惯。

不要等：

- 推上去以后
- GitHub Actions 报红以后
- 页面打不开以后

才去发现问题。  
更稳的顺序永远是：

1. 本地先改
2. 本地先构建
3. 构建通过再推

### 7.3 GitHub Pages 与静态导出类注意事项

#### 1. `BASE_PATH=/my-blog` 这件事不能忘

这个项目部署在 GitHub Pages 项目页场景下时，最关键的一件事就是：

```text
BASE_PATH=/my-blog
```

如果漏掉或处理不对，最常见的问题包括：

- 静态资源 404
- 页面跳转错位
- 搜索路径错误
- 图片路径不对

很多人看到页面 404 会先怀疑 Pages，其实根因往往是路径没统一处理好。

#### 2. 搜索、图片、导航链接特别容易在静态导出下出问题

这个项目后面专门修过不少静态路径问题，说明这不是理论风险，而是真实踩过的坑。  
最容易出问题的地方包括：

- 搜索表单跳转
- 搜索结果页链接
- 图片资源路径
- 导航门户卡片链接或图标路径

本质上都是同一类问题：

**子路径站点下，前端路径拼接非常容易重复拼、漏拼或者相对路径理解错。**

#### 3. GitHub Pages 静态模式下，安全响应头有平台边界

这一点很容易误判。

项目里 `next.config.js` 可以配置安全响应头，但在 GitHub Pages 这种纯静态托管模式下，它们并不会像 Node 托管那样真正生效。  
这不是代码没写对，而是平台能力边界决定的。

所以这类问题要先分清：

- 是代码逻辑问题
- 还是平台天然做不到

#### 4. GitHub Actions 通过，不代表所有页面路径都一定正确

构建通过说明站点“能产出”，但不代表每个链接、每张图、每个搜索入口都正确。  
特别是做过：

- 路由调整
- 资源迁移
- 搜索改造
- `basePath` 相关修改

以后，更要实际点开检查关键页面。

### 7.4 统计与第三方依赖类注意事项

#### 1. `Busuanzi` 不能当成强依赖

这个项目的 UV/PV 展示依赖前端第三方脚本。  
这种方案的优点是轻量，但缺点也很明显：

- 第三方脚本可能临时不可用
- 网络环境可能影响加载
- 它不是你自己完全可控的服务

所以页面层面必须有兜底值。  
不能把它写成“如果统计脚本挂了，首页就一整块空掉”。

#### 2. 第三方方案要看“够不够用”，不是只看“高级不高级”

这个项目在统计方案上没有优先走重型自建后端，而是先选了低维护成本路线。  
这其实是一种很现实的取舍：

- 当前更重要的是把内容站维护稳
- 不是先给自己引入一套新的长期运维负担

所以后面看这类依赖时，不能只从“功能强不强”看，还要从“维护成本是否值得”看。

### 7.5 文档同步与维护类注意事项

#### 1. README、更新记录、项目笔记三类文档要同步维护

这个项目一个很容易被低估的坑是：

- 代码已经改了
- 页面已经变了
- 但文档还停在旧状态

这会导致后面再回看时特别痛苦，因为你会分不清：

- 当前真实状态是什么
- 某篇文档写的是过去状态还是当前状态

所以这个项目后面最值得坚持的习惯之一就是：

- 改结构，补 README
- 改行为，补更新记录
- 项目阶段性变化明显时，补项目笔记

#### 2. 写项目文档时不要把“规划中”写成“已上线”

这个项目里有些内容属于后续优化方向，比如：

- 自定义域名迁移
- Cloudflare 接入
- 某些更进一步的运维能力

这些东西文档里可以写，但必须写清楚“当前状态”和“后续规划”的边界。  
不然时间一久，自己都会被旧文档误导。

#### 3. 文档最好写成“以后出问题能直接照着查”的形式

这个项目里比较实用的几篇文档，价值都不只是“记录做过什么”，而是：

- 出问题时能快速定位
- 以后迁移时能直接复用
- 回头交接时别人也能看懂

这也是为什么像排障清单、迁移手册这类文档特别有价值。
