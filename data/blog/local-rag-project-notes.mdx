# Local RAG 项目笔记：一个本地优先的网络安全 Hybrid RAG 小项目

这份笔记记录的是我现在这个 `local-rag` 项目到底做到了哪一步、它的主链路是怎么跑起来的、现在还缺什么、接下来应该优先做什么。

这篇不是论文，也不是正式技术文档，主要是为了以后我自己回头看时，不会只记得“我好像做过一个 RAG”，而是真的能说清：

1. 这个项目是什么
2. 它现在是怎么工作的
3. 它和一般“纯聊天”项目有什么不一样
4. 它下一步最值得补哪些能力

---

## 1. 先用一句人话说清楚：这个项目是什么

这个项目本质上是一个**本地优先的网络安全知识库问答助手原型**。

它不是那种单纯把问题丢给大模型、让模型凭参数记忆随便回答的聊天程序，  
而是一个带有 **RAG（检索增强生成）** 思路的小系统：

- 先查本地知识库
- 再查受限的网络资料
- 最后再让本地模型结合证据回答

所以它的核心不是“模型自己多聪明”，而是：

**想办法先找到资料，再让模型基于资料回答。**

---

## 2. 这个项目现在到底做到了哪一步

先说结论：  
这个项目已经不是空架子了，最小闭环其实已经跑通了。

当前已经实现的东西包括：

- 本地文档目录 `docs/`
- 文档切块
- 向量化
- Chroma 本地向量库
- 本地 Ollama embedding 模型
- 本地 Ollama chat 模型
- 本地知识检索
- 白名单联网检索
- 最终答案生成
- 简单人工评估

也就是说，现在这版已经能完成下面这条链路：

1. 把本地网络安全资料放进 `docs/`
2. 运行 `src/ingest.py` 建索引
3. 用 `src/chat.py` 提问
4. 系统先查本地，再查网络
5. 最后输出答案，并附上本地来源和网络来源

所以它现在已经是一个**CLI 版、单轮问答版、网络安全主题版的 Hybrid RAG Demo**。

---

## 3. 这个项目和普通聊天程序最大的区别

很多人一开始做大模型项目时，最容易做成“问一句，模型编一句”。

这个项目和那种纯聊天程序的区别，主要在这里：

### 3.1 它不是直接问模型

普通聊天程序往往是：

- 用户提问
- prompt 拼一下
- 模型直接回答

而这个项目是：

- 用户提问
- 先去本地向量库检索
- 再去网络白名单站点检索
- 把证据拼进 prompt
- 再让模型回答

所以模型在这里更像“最后的整合器”，不是唯一信息来源。

### 3.2 它强调“本地优先”

这个项目名字里最关键的词其实不是 `RAG`，而是 `local-first`。

意思就是：

- 本地资料优先
- 网络资料只是补充
- 如果本地和网络冲突，要明确说有差异
- 如果两边都不够，就承认证据不足

这和很多“能答就答”的演示项目很不一样。  
它更像是在练一个比较稳妥的问答习惯。

### 3.3 它不是开放式联网搜索

现在的网络搜索也不是完全放开，而是做了白名单限制。

允许的网站主要是网络安全相关来源，比如：

- OWASP
- MITRE ATT&CK
- HackTricks
- Microsoft Learn
- Splunk
- CrowdStrike
- HoneyDB
- Wireshark
- Wikipedia

这样做的好处是：

- 减少乱七八糟的网页噪声
- 更容易保持答案风格一致
- 对学习型项目来说更可控

## 4. 目录结构到底怎么理解

这个项目不大，但结构其实挺清楚。

```text
local-rag/
├── .env.example
├── README.md
├── requirements.txt
├── chroma_db/
├── docs/
├── eval/
└── src/
    ├── chat.py
    ├── config.py
    ├── ingest.py
    └── web_search.py
```

### 4.1 `docs/` 是本地知识库原料区

这里放的不是数据库表，而是原始知识材料。

现在的内容主要是网络安全学习型主题，比如：

- 蜜罐
- 渗透测试
- 红蓝对抗
- MITRE ATT&CK
- OWASP Web 安全测试
- 威胁狩猎
- 隐私计算

这些文档已经被整理成比较适合 RAG 的知识条目，而不是原始网页直接扔进来。

### 4.2 `src/ingest.py` 是建库脚本

这个文件负责做三件事：

1. 读取 `docs/` 下的文本
2. 把文档切块
3. 生成 embedding 并写入 Chroma

它是“知识库构建阶段”的主入口。

### 4.3 `src/chat.py` 是问答主入口

这个文件负责：

1. 接收问题
2. 查本地向量库
3. 决定是否联网补充
4. 拼 prompt
5. 调本地聊天模型生成答案
6. 打印答案和来源

它是“检索 + 生成阶段”的主入口。

### 4.4 `src/web_search.py` 是联网补充模块

这个模块做得比“随便搜一下网页”稍微认真一点。

它里面其实已经包含好几层逻辑：

- 用模型把问题改写成更适合搜索的英文查询
- 用 DuckDuckGo HTML 后端搜候选结果
- 做域名白名单过滤
- 给候选网页打分
- 抓网页正文
- 再把抓到的网页转成 Document

所以它不是一个简单的 `requests.get()` 小脚本，而是已经有一点“轻量检索策略”的味道了。

### 4.5 `src/config.py` 是统一配置入口

这个文件负责读取 `.env`，把项目里会调的参数统一集中起来，比如：

- Ollama 地址
- 聊天模型
- embedding 模型
- 文档目录
- 向量库目录
- chunk 大小
- 检索条数
- 是否开启网络搜索
- 网络搜索允许域名

也就是说，它把“项目行为怎么变”这件事，从代码里抽出来了一部分。

### 4.6 `eval/` 是最初级的评估区

这里已经有：

- `questions.txt`
- `results.txt`
- `results_subset_after_fix.txt`

这说明项目已经开始往“不是只跑通，还要看看答得怎么样”这一步走了。

虽然现在还比较手工，但方向是对的。

## 5. 这条 RAG 主链路到底是怎么跑起来的

这一节是整个项目最重要的部分。

### 5.1 第一步：加载本地文档

`ingest.py` 会递归读取 `docs/` 目录下支持的文件类型：

- `.md`
- `.txt`
- `.py`

如果文件：

- 不是文本
- 编码不对
- 内容是空的

就会跳过。

这一步的本质是：  
先把“我有哪些资料”整理成程序能理解的 `Document` 对象。

### 5.2 第二步：切块

文档不会直接整篇塞进向量库，而是会先切块。

当前默认参数是：

- `CHUNK_SIZE=800`
- `CHUNK_OVERLAP=150`

这样做的原因很直接：

- 文档太长，检索命中不稳定
- 切小以后更容易匹配到相关段落
- overlap 可以减少边界信息被切断的问题

### 5.3 第三步：向量化并写入 Chroma

切好的 chunk 会通过 Ollama embedding 模型：

- `qwen3-embedding:0.6b`

编码成向量，然后写入本地 Chroma 向量库。

所以 `chroma_db/` 本质上就是“把本地知识变成可相似度检索的向量索引”。

### 5.4 第四步：回答问题时先查本地库

当用户提问时，`chat.py` 会先对本地向量库做相似度检索。

它不是简单地拿前 `k` 条，而是还有一点过滤逻辑：

- 先看总 chunk 数
- 拿一批候选
- 按得分阈值过滤
- 控制和最佳分数的差距
- 尽量避免同一来源文件重复占满结果

这说明作者已经开始意识到：  
“检索结果不是越多越好，乱召回反而会污染后面的生成。”

### 5.5 第五步：再决定要不要联网补充

如果启用了 `ENABLE_WEB_SEARCH=true`，项目会并发做网络搜索。

网络搜索的逻辑大概是：

1. 先把问题改成短一点的英文搜索词
2. 用 DuckDuckGo 找候选链接
3. 按白名单过滤域名
4. 给候选页面打分
5. 抓正文
6. 选出最合适的少量网页

也就是说，这里不是“全网乱搜”，而是“带约束的辅助检索”。

### 5.6 第六步：把本地证据和网络证据一起塞给模型

最后 prompt 会明确告诉模型：

- Local Context 是主证据
- Web Context 只是补充
- 如果两者冲突，要明确说有差异
- 如果证据不够，不要瞎编

这一步非常关键。  
因为 RAG 项目最容易失败的地方，不是“没检索到”，而是“检索到了但 prompt 约束不够，模型还是乱说”。

### 5.7 第七步：输出答案和来源

最后结果会打印：

- 回答正文
- 本地来源
- 网络来源

这一点虽然简单，但其实是一个很重要的好习惯：

- 不只看回答
- 还要看它到底是从哪里来的

## 6. 现在这版项目的优点在哪里

我觉得当前版本最大的优点不是“功能特别多”，而是**路线很干净**。

### 6.1 它把 RAG 最核心的闭环做出来了

也就是：

- 数据准备
- 切块
- 向量库
- 检索
- 生成
- 来源输出
- 初步评估

很多人做到一半就散了，这个项目至少已经把主线打通。

### 6.2 它的主题收得比较稳

这不是一个“什么都想问”的万能助手，  
而是比较明确地聚焦在：

- 网络安全学习型知识问答

这个聚焦很重要。  
因为越聚焦，越容易：

- 控制资料质量
- 控制白名单来源
- 控制回答风格
- 评估到底答得好不好

### 6.3 它比纯本地 RAG 多了一点“联网补全”的现实感

很多教学项目只做本地知识库。  
这个项目往前多走了一步：

- 本地知识优先
- 网络知识补充

这更接近真实使用时的需求，因为本地资料不可能永远全覆盖。

### 6.4 它有评估意识

`eval/` 虽然还不高级，但已经说明一个很重要的方向：

> 不是只看“它能不能回答”，而是要看“它回答得稳不稳、是否偏题、是否缺证据”。

这一步很值得保留。

## 7. 现在这版项目还缺什么

虽然现在已经能跑，但它还明显是一个“第一版原型”。

### 7.1 只有 CLI，没有图形界面

现在的交互还是：

```bash
python src/chat.py "你的问题"
```

这非常适合验证主链路，但还不适合后面真正展示或长期使用。

### 7.2 只有单轮，没有会话记忆

当前 README 里也明确说了，这是：

- CLI-only
- single-turn

所以它现在更像“单次问答工具”，还不是一个能连续交流的知识助手。

### 7.3 本地知识库规模还比较小

现在 `docs/` 里的资料是一个不错的起点，但量级还比较有限。

问题在于：

- 主题还不够宽
- 深度也不够均衡
- 内容粒度偏“学习概览”

所以遇到更细的问题时，系统还会比较依赖联网补充。

### 7.4 检索还没有 rerank

当前本地检索主要还是：

- 向量相似度
- 分数阈值过滤
- 简单去重

这已经比“裸 top-k”好一些了，但还没有做到：

- 交叉编码器重排
- 多路召回
- query expansion
- 混合 BM25 + vector

所以更复杂的问题下，召回质量还有提升空间。

### 7.5 评估还比较手工

现在的评估更像：

- 准备一批问题
- 手工跑结果
- 人眼看回答

这对第一版足够，但后面如果想持续优化，就需要更稳定的评估方式。

### 7.6 还没有真正的“产品感”

现在这版更像工程练手和原型验证，还不是完整产品。

比如还缺：

- 多轮对话体验
- 更好的错误提示
- 更明确的来源展示
- 历史记录
- 文档上传入口
- 用户侧操作界面

## 8. 这个项目和 Datawhale 那条学习路线怎么对上

我觉得这个项目和 Datawhale 的“个人知识库助手”路线是能对上的，而且已经走到了中间阶段。

Datawhale 那条线可以大致理解成：

1. 大模型应用开发入门
2. 知识库搭建
3. RAG 应用构建
4. 应用部署
5. 评估与迭代

而我现在这个项目大致对应到：

- **知识库搭建**：已经做了
- **RAG 应用构建**：已经做了最小闭环
- **应用部署**：还没正式做界面化部署
- **评估与迭代**：刚起步

所以当前最像的状态其实是：

> 已经从“会调用模型”走到了“会做一个最小知识库问答系统”，下一步该从 Demo 走向更完整的应用。

这里最值得借鉴 Datawhale 的，不是照着它抄实现，而是借它那条课程路线提醒自己：

- 不要停在“能跑”
- 要继续往“能用、能评估、能迭代”推进

## 9. 接下来应该干什么

这一节最重要。  
因为这个项目现在不是“没做出来”，而是“已经做出来一部分，但还没继续往下补”。

如果让我按收益和顺序来排，我会这样往下走。

### 9.1 第一优先级：把本地知识库内容补强

现在最该做的不是先换模型，而是先把知识库质量提上去。

优先补的方向可以是：

- Web 安全基础
- 常见漏洞原理
- 常见防御手段
- 安全运维与日志分析
- 取证、应急响应、恶意代码基础

原因很简单：

- RAG 的上限很大程度取决于资料质量
- 本地资料太薄时，模型会更依赖网络补充
- 这样“本地优先”的优势就发挥不出来

### 9.2 第二优先级：做更系统的评估集

现在 `eval/questions.txt` 已经是个好起点，但还不够。

接下来可以把问题分层整理成几类：

- 事实定义类
- 概念对比类
- 步骤解释类
- 边界辨析类
- 超纲拒答类

这样以后每次改：

- chunk 参数
- 检索策略
- prompt
- 模型

都能更稳定地比较效果。

### 9.3 第三优先级：把来源展示做得更细

现在已经会打印来源了，但还可以继续升级。

我觉得比较值得补的是：

- 显示命中的 chunk 编号
- 显示更短的证据摘录
- 明确本地证据和网络证据分别支撑了哪句话

这样做的好处是，后面排查“为什么答成这样”会轻松很多。

### 9.4 第四优先级：做一个最小网页界面

现在 CLI 非常适合开发，但不适合展示。

下一步很适合加一个最小前端，比如：

- `Streamlit`
- `Gradio`

先不要想得太重，最小目标可以只是：

- 一个输入框
- 一个回答区
- 一个来源区
- 一个开关：是否启用网络搜索

这样项目就会从“脚本”更像“应用”。

### 9.5 第五优先级：补多轮对话和会话状态

如果要往“助手”继续靠，多轮对话迟早要加。

但我觉得它不该排第一优先级。  
因为如果单轮证据链都还没做稳，多轮只会把问题叠得更复杂。

更合理的顺序是：

1. 先把单轮检索和回答做好
2. 再加会话记忆
3. 再考虑历史上下文怎么进入检索

### 9.6 第六优先级：优化检索，而不是第一时间换更大的模型

很多人一遇到答不好，就想先换更大的模型。  
但对这个项目来说，当前更值得补的是检索层。

可以逐步尝试：

- query rewrite 优化
- 混合检索（关键词 + 向量）
- rerank
- 更好的 chunk 策略
- 文档级 / chunk 级去重

因为很多回答问题，本质上不是模型不会，而是前面资料没找准。

### 9.7 第七优先级：增加拒答和越权问题处理

当前项目已经有“证据不足就别乱说”的意识，这是对的。  
下一步可以把这个意识做得更明确一些，比如：

- 非网络安全问题时更稳地拒答
- 高风险操作细节时更谨慎
- 证据冲突时更明确说出差异来源

这会让整个项目更像一个可靠助手，而不是一个会胡乱补全的聊天壳子。

## 10. 如果让我现在就继续做，我会按这个顺序推进

为了避免后面又变成“想做很多，但不知道先动哪一个”，我给自己排一个更具体的执行顺序。

### 第 1 步：补知识库内容

目标：

- 先把 `docs/` 扩到一个更像样的网络安全学习库

最直接的动作：

- 每个主题先补 2 到 4 篇高质量整理文档
- 保持格式统一
- 一篇文档只讲一个主题

### 第 2 步：重做评估题集

目标：

- 从“零散问题”变成“分类型题库”

最直接的动作：

- 把 `eval/questions.txt` 改成按类别维护
- 单独加一组“证据不足时应拒答”的问题

### 第 3 步：补最小网页界面

目标：

- 让它从命令行 Demo 变成可以展示的原型

最直接的动作：

- 先用 Streamlit 做一个单页版
- 先不追求好看，先追求可用

### 第 4 步：优化检索

目标：
