这篇文章是对当前站点现状的一个澄清，也是我后续优化路线的规划文档。

先把最重要的一句话放在最前面：

**当前这个网站已经部署在 GitHub Pages 上，但还没有接入 Cloudflare。**

也就是说，当前线上结构是：

- 代码托管：GitHub
- 自动构建：GitHub Actions
- 静态托管：GitHub Pages
- 访客统计展示：Busuanzi

暂时还没有这些能力：

- 自定义域名
- Cloudflare DNS
- Cloudflare CDN / 缓存策略
- Cloudflare 安全规则
- Cloudflare 访问层优化

所以如果后面看到我提到 `Cloudflare`，需要明确区分两种语境：

- 一种是“当前网站已经接了 Cloudflare”
- 另一种是“后续如果我要继续优化，Cloudflare 是一个可选方案”

就这个项目当前状态而言，正确的说法只能是第二种。

## 一、为什么我要专门澄清这件事

原因很简单：

- 技术文章里最怕把“未来规划”写成“当前架构”
- 一旦表述不清，以后自己回头看也会混乱
- 如果别人参考我的文章，也可能误以为我已经做了自定义域名和 Cloudflare 接入

所以这篇文章的目标非常明确：

1. 说清楚当前站点真实运行方式
2. 说清楚为什么我后面会考虑 Cloudflare
3. 说清楚如果真的要接，应该怎么一步一步做

## 二、当前站点到底是怎么跑起来的

现在这个网站的真实链路其实很简单：

1. 我在本地写文章和改代码
2. 提交到 GitHub 仓库
3. GitHub Actions 自动构建
4. Next.js 以静态导出方式生成 `out/`
5. GitHub Pages 发布静态内容
6. 用户通过 GitHub Pages 地址访问站点

当前访问地址是：

```text
https://laixian805-tech.github.io/my-blog
```

这个地址本身就说明了两件事：

- 它是 GitHub Pages 项目页
- 它还没有切到自定义域名

## 三、为什么现在没有接 Cloudflare

不是因为不能接，而是因为当前阶段我更关注“先跑通、先稳定更新、先把内容写起来”。

如果太早引入 Cloudflare，虽然能力会更多，但也会多出一批新的配置工作：

- 域名购买和管理
- DNS 解析
- SSL 证书细节
- 缓存策略
- 页面规则
- 安全规则
- 跳转规则

对一个当前以内容发布为核心目标的个人博客来说，第一阶段更重要的是：

- 网站能稳定访问
- 更新流程足够简单
- 内容结构先搭起来

所以我现在选择的是：

- 先用 GitHub Pages 直接托管
- 先把文章、搜索、Projects、Security 这些内容模块做好
- 等内容站真正进入长期维护阶段，再处理域名和 Cloudflare 这类“第二阶段优化”

## 四、为什么后面仍然可能考虑 Cloudflare

虽然现在没接，但从长期来看，Cloudflare 仍然是很值得考虑的一层。

主要原因有这些。

### 1. 更适合长期品牌化

如果以后我想让这个站点更像一个长期品牌站，而不是一个默认的 GitHub Pages 地址，那么自定义域名几乎是必做项。

例如：

```text
blog.example.com
example.com
notes.example.com
```

当开始使用自定义域名以后，Cloudflare 就会变得很有价值，因为它通常会成为：

- DNS 管理入口
- HTTPS 与证书的辅助层
- 缓存和访问优化入口
- 安全规则的统一入口

### 2. 访问和缓存策略更灵活

GitHub Pages 能托管静态页面，但它不是一个专门为“细粒度流量控制”设计的平台。

如果后续我希望：

- 某些资源缓存更久
- 某些页面更快刷新
- 某些路径做特殊规则

那么 Cloudflare 提供的灵活度通常会更高。

### 3. 安全能力更容易补强

对于个人博客来说，不一定一开始就需要很多安全配置，但后面如果站点逐渐稳定运行，还是会希望多一些基本保护能力，例如：

- 基础防护
- 速率限制
- 机器人与异常流量处理
- 某些路径的访问规则

这些能力在 Cloudflare 上通常比较容易统一配置。

### 4. 后面切换托管平台更平滑

如果未来我从 GitHub Pages 换到别的平台，例如：

- Vercel
- Netlify
- Cloudflare Pages
- 自己的服务器

那么只要域名和 DNS 层管理得好，用户感知通常会更平滑。

这也是为什么我把“自定义域名 + DNS 层能力”看成长期可迁移性的一部分。

## 五、如果以后真的要接，我会按什么顺序做

我不建议一上来就同时改很多项。

更稳的做法是按顺序逐步推进。

### 第一阶段：先保持 GitHub Pages 正常运行

这一阶段先不要动太多基础设施，只做这几件事：

- 保持站点持续可访问
- 保持 GitHub Actions 发布稳定
- 保持内容持续更新
- 保持 SEO 地址配置统一

这一步的目标不是“功能更多”，而是“先把基础盘子做稳”。

### 第二阶段：购买并绑定自定义域名

如果我要从 GitHub Pages 默认域名升级到自定义域名，通常会先做：

1. 购买域名
2. 选择主域名或子域名方案
3. 在 GitHub Pages 中配置 Custom domain
4. 验证 HTTPS 和跳转是否正常
5. 更新站点的 `SITE_URL`

这里最关键的是先统一“最终正式地址”。

如果正式地址没定下来，后面 SEO、分享图、RSS、canonical 都会反复变。

### 第三阶段：把域名接到 Cloudflare

这一步才是 Cloudflare 真正进场的时候。

一般会涉及：

1. 将域名 NS 切换到 Cloudflare
2. 在 Cloudflare 中配置 DNS 记录
3. 确保自定义域名仍然正确指向 GitHub Pages
4. 检查 HTTPS 是否正常
5. 检查重定向和缓存是否正常

到这一步，Cloudflare 才算真正成为这个网站访问链路的一部分。

### 第四阶段：逐步增加优化规则

等域名和访问稳定后，再考虑逐步加能力，例如：

- 更合理的缓存策略
- 重定向规则
- 安全头补充
- 基础访问控制
- WAF / Rate Limit 之类的能力

这里最重要的原则是：

**一次只加一小层，不要一次性把所有规则都堆上去。**

否则后面出问题时很难定位到底是哪一层配置导致的。

## 六、如果我要写成一套具体实施方案，技术上会涉及什么

为了以后自己执行更顺手，我把技术实现面也先拆一下。

### 1. 域名层

这一层要解决的问题是：

- 我最终想让用户访问哪个地址
- 是根域名还是子域名
- 域名在哪个平台购买
- DNS 谁来托管

常见选择是：

- 根域名做主站
- `blog.` 子域名做博客

这不是纯技术问题，也和个人品牌表达方式有关。

### 2. GitHub Pages 层

这一层主要关注：

- Custom domain 配置
- GitHub Pages 的 HTTPS 生效
- 发布路径是否正确
- 站点是否还需要 `basePath`

这里有一个非常关键的问题：

如果以后切到独立自定义域名，例如：

```text
https://blog.example.com
```

那么当前的项目页路径 `/my-blog` 方案可能就不再适合继续保留。

到那个时候，通常要重新评估：

- `BASE_PATH` 是否还需要
- `SITE_URL` 是否切换
- canonical 是否更新
- RSS / sitemap 是否更新

### 3. Cloudflare 层

如果引入 Cloudflare，我主要会把它当成这些能力集合：

- DNS
- 缓存
- 安全
- 流量入口控制

但注意一点：

当前这个网站还没有这些配置，所以现在不能把 Cloudflare 写成“现有架构的组成部分”。

### 4. Next.js 配置层

如果后面切域名或路径策略，我最需要关注的是这些配置：

- `data/siteMetadata.js` 里的 `siteUrl`
- GitHub Actions 里的 `SITE_URL`
- `BASE_PATH`
- `NEXT_PUBLIC_LAST_UPDATED`
- sitemap / robots / canonical 的最终绝对地址

这些东西如果没改统一，很容易出现：

- 页面能打开，但 canonical 还是旧地址
- RSS 地址不对
- 分享图链接还是旧地址
- sitemap 里保留旧路径

## 七、如果以后要更稳定，我真正优先级最高的事情是什么

虽然 `Cloudflare` 很有吸引力，但如果只谈优先级，我会更看重下面这几个点。

### 1. 正式域名尽快固定

比起“接不接 Cloudflare”，更高优先级的是：

- 最终正式域名是什么
- 后面是否长期不再改

地址一旦稳定，SEO 和外部引用才容易稳定下来。

### 2. 发布流程要足够简单

对个人网站来说，流程越简单，越稳定。

如果一个站点每次更新都要经过很多平台和手动配置，它长期维护的成本会明显变高。

所以我现在很认可的一条原则是：

**第一阶段先让发布链路最短。**

也就是：

- GitHub
- GitHub Actions
- GitHub Pages

这条链已经足够支持当前博客稳定运行。

### 3. 站点内容和结构要先成熟

如果内容结构还在频繁变化，太早投入很多流量层和安全层优化，收益并不一定高。

更现实的顺序通常是：

1. 先把内容体系搭起来
2. 先让项目页、导航页、搜索页稳定
3. 再去做更长期的访问层优化

## 八、如果以后真的要做，我建议自己的最小落地路线

为了避免把事情想得太大，我给自己写一个最小落地版本。

### 方案 A：只加自定义域名，不加 Cloudflare

这是最轻的一条路线。

适合场景：

- 只是想把 GitHub Pages 地址换得更专业
- 还不想额外维护太多 DNS 和规则配置

优点：

- 改动小
- 风险低
- 收益直接

### 方案 B：自定义域名 + Cloudflare 基础 DNS

这是我觉得比较平衡的一条路线。

适合场景：

- 想要更专业的域名
- 想把 DNS 和后续优化能力掌握在自己手里
- 但还不急着上很多高级规则

优点：

- 后续扩展空间大
- 可以先简单接入，后续再慢慢加能力

### 方案 C：自定义域名 + Cloudflare + 更系统的规则优化

这是更长期的路线。

适合场景：

- 内容量更大
- 访问更稳定
- 开始更关注安全、缓存、监控和品牌化

但这条路线不适合作为第一步，因为复杂度更高。

## 九、我给未来自己的结论

当前这个网站的真实状态很明确：

- 已上线 GitHub Pages
- 已接入 Busuanzi
- 还没有接入 Cloudflare

所以任何涉及 `Cloudflare` 的内容，在今天都应该被表述为：

- 后续优化方向
- 可选升级路线
- 未来实施方案

而不是：

- 当前线上架构
- 当前已经使用的服务

如果以后我真的接入了 Cloudflare，我会再补一篇新的实战记录，专门写清楚：

- 域名怎么选
- DNS 怎么配
- GitHub Pages 怎么配合
- `SITE_URL` 和 `BASE_PATH` 怎么调整
- 接入后实际提升了什么

在那之前，最准确的描述始终是：

**这个网站当前没有部署 Cloudflare，Cloudflare 只是后续优化方案。**
