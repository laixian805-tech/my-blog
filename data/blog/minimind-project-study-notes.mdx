# MiniMind 项目学习笔记：从跑通到真正看懂

这份笔记不是只记录“我执行了哪些命令”，而是把我在 `MiniMind` 项目里**实际做过的步骤、每一步为什么做、做完后看到了什么、背后的原理是什么**都串起来。

目标不是背命令，而是以后回看时，能快速想起：

1. 我当时到底做了什么。
2. 为什么先做这一步，而不是做别的。
3. 每一步解决的是什么问题。
4. 哪些现象是正常的，哪些才是异常。
5. 这个项目的训练主线到底是什么。

---

## 第 0 部分：先用人话说清楚，这个项目是什么

`MiniMind` 本质上是一个**面向大模型初学者的“从 0 到 1”教学型项目**。

它最有价值的地方不在于“参数多大”，而在于它把大模型里最重要的几条主线尽量白盒化了：

- 模型结构是怎么搭起来的
- 数据是怎么喂进去的
- `pretrain` 和 `SFT` 有什么区别
- 权重是怎么保存、怎么加载、怎么推理的
- LoRA 这种轻量微调到底在改什么

如果把它和现实里的学习过程类比：

- `Pretrain` 像先让学生大量读书，学语言和常识
- `SFT` 像再给学生做“客服培训”，学怎么按提问方式回答
- `LoRA` 像给一个已经会说话的人加一个“人设补丁”或“领域补丁”

这个项目特别适合我这种场景：

- 不是想直接“用现成大模型做产品”就结束
- 而是想借一个能跑起来的项目，真正理解训练过程和代码主线

## 第 1 部分：这次我在 MiniMind 里真正走过的主线

这次我不是一下子学完所有东西，而是按下面这条主线推进的：

1. 先读 `README`，弄清项目推荐的学习路线
2. 先跑通“第一种部署方式”，也就是现成模型推理
3. 再进入“模型训练”主线
4. 下载最小训练数据集
5. 解决 WSL 路径和 GPU / PyTorch 兼容问题
6. 跑 `pretrain`

8. 跑 `full_sft`
9. 测试 `full_sft` 输出
10. 回头读训练脚本、数据集代码、模型代码
11. 再进入 LoRA，做“给模型加人设/风格”的实验

这条顺序是很重要的，因为它不是乱试，而是：

**先把主线跑通，再回头解释为什么。**

如果一上来就啃 `Attention`、`RoPE`、`MoE`、`DPO`、`GRPO`，很容易信息量爆炸，最后什么都没真正抓住。

## 第 2 部分：第一步，我先读 README，确认项目推荐路线

### 我做了什么

我先读了项目里的：

- `README.md`
- `README_en.md`
- 老版本 `MiniMind2_real/README.md`

重点不是看完所有内容，而是确认：

- 项目最核心的训练入口在哪里
- 训练最推荐的数据组合是什么
- 新手最该先走哪条路线

### 为什么这样做

因为训练项目最怕一上来就做错路径。

像 MiniMind 这种仓q

- LoRA
- DPO
- PPO / GRPO / CISPO
- Agentic RL
- Tool Use
- 推理模板

如果不先读 README，很容易误以为：

- 都要学
- 都要跑
- 或者先跑 LoRA 就等于q了大模型训练

但项目主线其实非常清楚：

- **最小闭环**：`pretrain_t2t_mini + sft_t2t_mini`
- **训练主线**：`Pretrain -> SFT -> Eval`

### 当时读 README 得到的关键结论

1. 新手先走最小闭环，不要一上来碰 RL / Agent / MoE
2. 推荐先下载：
   - `pretrain_t2t_mini.jsonl`
   - `sft_t2t_mini.jsonl`
3. 训练分两阶段：
   - 先 `train_pretrain.py`
   - 再 `train_full_sft.py`
4. `full_sft` 默认接 `pretrain` 权重继续训练

### 这一步背后的学习原则

**先确认主线，再动手。**

以后看任何大模型项目，第一件事都应该不是“先运行最长的命令”，而是：

- 项目主线是什么
- 作者推荐初学者先跑哪条线
- 哪些是核心步骤，哪些是扩展步骤

## 第 3 部分：我先跑通了现成模型推理

### 我做了什么

我先走了 README 里的第一种部署 / 推理方式，把现成模型跑起来。

这一步的目标不是训练，而是先确认：

- 项目能正常加载模型
- 推理脚本 `eval_llm.py` 能跑
- 我对 `weight` / `load_from` / tokenizer / 输出形式有一个直观感觉

### 为什么先做这个

因为如果连现成模型都跑不通，就不适合马上进入训练。

先跑推理有几个好处：

1. 先知道“训练完以后我最终会得到什么”
2. 先熟悉 `eval_llm.py` 的使用方式
3. 先确认基础环境（Python、依赖、GPU）至少能支持推理

### 这一步的意义

它让我先有一个“终点长什么样”的感受：

- 最后模型不是一堆文件，而是能实际回答问题的程序
- `.pth` 权重不是拿来打开看的，而是拿来被程序加载推理的

这会帮助后面理解：

- 为什么训练最后保存的是 `out/*.pth`
- 为什么 `eval_llm.py` 要负责加载权重和 tokenizer

```

```

因为训练最开始就需要放数据集文件。

## 第 6 部分：训练前我排查了 GPU / PyTorch 兼容问题

### 一开始遇到的问题

最开始环境里虽然有 PyTorch，但版本和我的显卡不匹配。

当时查到的信息大致是：

- 旧环境里 `torch 2.6.0 + cu124`
- GPU 是 `RTX 5070 Laptop GPU`
- 旧 PyTorch 对 `sm_120` 支持不完整

结果就是：

- 看起来 `cuda.is_available()` 可能为真
- 但实际训练并不稳，或者明确警告不兼容

### 后来我做了什么

最后环境确认到了：

- `torch 2.11.0+cu128`
- `cuda_available=True`
- `arch_list` 里已经包含 `sm_120`

之后才能真正用 GPU 跑训练。

### 为什么这一步很重要

因为“大模型训练能不能开始”，第一道门槛往往不是代码，而是：

- GPU 能不能被 PyTorch 正确认出来
- CUDA 架构兼不兼容
- torch 版本和显卡新架构是否匹配

### 新手在这里最容易踩的坑

很多人看到：

```python
torch.cuda.is_available() == True
```

就以为环境没问题。

其实还不够。

更关键的是：

- 你的 GPU 架构是否被当前 torch 编译支持
- 训练时会不会报兼容性警告
- `torch.cuda.get_arch_list()` 是否包含你的架构

### 这一步的学习原则

**能看到 GPU，不等于能稳定训练。**

训练前最好确认：

1. `torch.__version__`
2. `torch.version.cuda`
3. `torch.cuda.is_available()`
4. `torch.cuda.get_device_name(0)`
5. `torch.cuda.get_arch_list()`

## 第 7 部分：我正式开始跑 `pretrain`

### 我做了什么

我使用 GPU 跑了 `train_pretrain.py`。

为了适配我自己的 8GB 显卡，没有直接用 README 里理想机器的默认感觉，而是用了更保守的参数，例如：

- `--device cuda:0`
- `--batch_size 4`
- `--accumulation_steps 8`
- `--num_workers 2`
- `--max_seq_len 256`

### 为什么要这样调

因为我的机器不是 README 里举例的 24GB 3090。

如果直接照搬很激进的设置，很容易：

- OOM
- 训练速度异常
- 新手误以为“项目跑不通”

### 这一步背后的原理

`pretrain` 做的事情非常纯粹：

**给模型看前文，让它预测下一个 token。**

也就是：

- 输入一段文本
- 模型预测接下来该出现什么 token
- 用真实 token 算 loss
- 再反向传播更新参数

### 当时我在日志里看到了什么

训练日志里会打印：

- `loss`
- `logits_loss`
- `aux_loss`
- `lr`
- `epoch_time`

其中我最先学会看的其实只有 3 个：

1. `loss` 有没有整体往下走
2. `aux_loss` 是否为 0（dense 模型正常）
3. 没有报 OOM 或设备错误

### `epoch_time` 是什么意思

一开始我误以为那是“总共训练时间”，后来搞清楚：

它在这个脚本里其实更接近“**剩余时间估算**”。

也就是说：

- 不是说还要重新训练一轮
- 而是按当前速度，估计这一轮还剩多少分钟

### 这一步给我的最重要收获

我第一次真正看到了：

- 大模型训练不是一个神秘黑盒
- 本质上就是反复让模型做“下一个 token 预测”
- loss 的变化不是一条直线，而是有波动但整体下降

## 第 8 部分：我第一次测试 `pretrain` 权重

### 我做了什么

训练保存出 `pretrain_768.pth` 后，我运行：

```bash
python eval_llm.py --weight pretrain
```

然后手动问模型：

- 为什么天空是蓝色的？
- 什么是机器学习？
- 推荐三种中国美食。
- 你是谁？

### 为什么要马上测试

因为这一步的目标不是“冲效果”，而是验证：

**pretrain 到底让模型学到了什么。**

### 我看到的现象

非常典型：

- 模型已经能说很长一段话
- 语言上看起来像在解释
- 但事实混乱、跑题严重、一本正经胡说

比如问“为什么天空是蓝色的”，它会从天空一路扯到太阳系、地球距离、海王星、火星……

### 这个现象为什么是正常的

因为 `pretrain` 学的是：

- 语言接续
- 文本风格
- 知识碎片

它学会的是“像样地继续写”，不等于学会“稳定、收敛地回答用户问题”。

### 这一步的核心结论

`Pretrain` = 学会说话  
但还不等于 = 会好好回答

这是我理解 `SFT` 必要性的关键转折点。

## 第 9 部分：我开始跑 `full_sft`

### 我做了什么

我基于 `pretrain_768.pth` 继续跑了：

```bash
python trainer/train_full_sft.py ...
```

核心思路是：

- 模型结构不变
- 起点不再是 `none`
- 而是加载刚才训练出的 `pretrain` 权重

### 为什么 `SFT` 要接 `pretrain`

因为：

- `pretrain` 提供语言和知识底座
- `SFT` 在这个底座上继续学“助手行为”

如果没有 `pretrain`，直接做 `SFT`，就像一个还没学好基础知识的人直接去做客服培训，效果会很差。

### `SFT` 的原理到底是什么

它本质上还是 `next token prediction`，  
但重点变成了：

**只训练 assistant 回复部分。**

也就是说：

- 模型先看 user / system / tool 上下文
- 真正算 loss 的重点是 assistant 后续该怎么说

### 这一步数据和 `pretrain` 最大区别

`pretrain` 数据：

```json
{"text": "..."}
```

`sft` 数据：

```json
{"conversations": [...]}
```

所以：

- `pretrain` 更像学“文章续写”
- `sft` 更像学“对话应答”

### 训练日志里我注意到的一个关键点

`SFT` 的学习率明显比 `pretrain` 小。

这也符合原理：

- `pretrain` 从零学，需要步子大一点
- `sft` 是在已有模型上精修，所以学习率更小、更稳

## 第 10 部分：我第一次测试 `full_sft`

### 我做了什么

训练保存出 `full_sft_768.pth` 后，我运行：

```bash
python eval_llm.py --weight full_sft
```

还是用同样问题去测：

- 为什么天空是蓝色的？
- 什么是机器学习？
- 推荐三种中国美食。
- 你是谁？

### 我看到的现象

和 `pretrain` 比起来，输出有了明显变化：

- 更像在扮演“助手”
- 更容易出现“您好，我是……”
- 更像在按问答格式组织输出

但问题也还很明显：

- 还是会跑题
- 还是会胡说
- 有时会出现 `</think>` 这类模板泄露

### 为什么会这样

因为这次实验里：

- `pretrain` 底座本来就还不够强
- `sft` 也没有训到非常充分
- 模型已经学会了“像助手说话”
- 但还没学稳“说得对”

### 这一步帮我真正建立的认知

1. `pretrain` 负责“会不会续写”
2. `sft` 负责“会不会按用户意图回答”
3. 如果底座弱，后面的 `sft` 也容易变成“格式正确地胡说”

这条因果链特别重要。

## 第 12 部分：我当时是怎么理解训练代码的

### 1. 训练循环本质上在做什么

无论 `pretrain` 还是 `sft`，训练循环大致都在做：

1. 从 `DataLoader` 里拿一批样本
2. 放到 GPU
3. 前向计算，得到 `loss`
4. 反向传播
5. 梯度累计若干步后更新参数
6. 打印日志
7. 定期保存权重

### 2. `loss` 的本质是什么

`loss` 不是神秘指标，它本质上就是：

**模型预测下一个 token 有多不准。**

### 3. 为什么是 `logits[..., :-1]` 对 `labels[..., 1:]`

因为当前位置负责预测“下一个位置”的 token。

一句话记忆：

**当前位置的预测，对齐下一个位置的标准答案。**

### 4. 为什么要 `accumulation_steps`

因为显存不够时，不能直接上大 `batch_size`，就可以：

- 每次喂小 batch
- 梯度先累计
- 累计若干步后再更新一次

这相当于“用多次小 batch 模拟一次更大的有效 batch”。

### 5. 为什么 `grad_clip` 很重要

它像训练时的“防失控保险丝”，防止某一步梯度太大把模型一下推飞。

## 第 13 部分：我后来真正理解了 `SFTDataset.generate_labels()`

这是我认为整个聊天模型学习里最关键的点之一。

### 它到底在做什么

它会先把整段对话都设成：

```python
-100
```

然后只把 assistant 回复那一段的标签改成真实 token。

再配合：

```python
ignore_index=-100
```

最终效果就是：

**只有 assistant 回复部分参与 loss。**

### 为什么要这样做

因为聊天模型不是在学“复述整段对话”，而是在学：

“看到用户和上下文之后，assistant 应该怎么回复。”

如果把 user、system、tool 全部也拿来算 loss，模型就会更像在“背整段剧本”，而不是学会针对性回答。

### 这一步对我的帮助

它让我彻底明白：

- `SFT` 不只是“继续训练”
- 它本质上是“改变标签区域”

这个理解特别重要。

---

## 第 14 部分：我后来是怎么理解整个模型前向路径的

一开始看 `model_minimind.py` 特别容易晕，后来我把它压成一条最小路线：

```text
token ids -> embedding -> 多层 block -> hidden states -> lm_head -> logits -> 下一个 token
```

### 1. `Embedding`

把 token id 变成高维向量。

### 2. `MiniMindBlock`

每层做两件事：

- Attention：看上下文
- FeedForward：自己再加工一下

### 3. `lm_head`

把隐藏向量映射回词表分数，也就是 `logits`。

### 4. `generate`

不是一次性写完整句，而是：

- 预测一个 token
- 拼回输入
- 再预测下一个
- 一直重复

### 我最后得出的最重要结论

大语言模型看起来像在“思考”，本质上是在一个很强的上下文建模系统里，**不断预测下一个 token**。

## 第 15 部分：做完主线后，我开始学 LoRA

### 为什么这时才学 LoRA

因为 LoRA 不是大模型训练主线的起点，而是：

**在已经有一个基础模型后，再做轻量定制。**

如果在还没弄懂：

- `pretrain`
- `sft`
- `eval`

之前就跳 LoRA，很容易误以为 LoRA 就等于“训练大模型”。

其实不是。

LoRA 更像：

- 给已有模型加一个补丁
- 让它偏向某种风格、身份或小领域能力

### 我当时实际做了什么

我没有直接上医疗或复杂垂域，而是先做了一版**身份 / 风格 LoRA**。

我新建了：

```bash
dataset/lora_identity_starter.jsonl
```

里面放的是一组很小的“人设 + 回答风格”样本，比如：

- 你是谁？
- 你的回答风格是什么？
- 如果你不确定答案怎么办？
- 如果用户是新手，你会怎么解释？

然后基于当前 `full_sft_768.pth` 训练出：

```bash
out/lora_identity_starter_768.pth
```

### 为什么先做人设 LoRA

因为它最容易观察。

如果一开始做知识型 LoRA，新手不容易判断：

- 这是基础模型本来就会
- 还是 LoRA 真学会了

但人设 / 风格变化很容易看出来：

- 说话方式有没有变
- 自我定位有没有变
- 解释习惯有没有偏移

## 第 16 部分：LoRA 的原理，我后来是怎么理解的

### 一句话

LoRA 的本质是：

**冻结原模型主体，只在部分线性层旁边加一个很小的低秩增量，然后只训练这个增量。**

### 为什么要这么做

因为如果直接全参数微调：

- 显存更贵
- 训练更慢
- 保存成本更高
- 更容易把原来的通用能力训坏

LoRA 解决的问题就是：

“能不能不动整个大模型主体，只加一个小补丁，就让它朝某个方向偏过去？”

### 在这个项目里的体现

`model_lora.py` 里核心做了几件事：

1. 定义 LoRA 小模块：`A` 和 `B`
2. 把它挂到原始线性层旁边
3. 前向变成：

```text
原输出 + LoRA 输出
```

4. 在 `train_lora.py` 里冻结非 LoRA 参数，只训练 LoRA 参数

### 我当时真正看到的数据

训练时打印出来大概是：

- 总参数约 `64.3M`
- LoRA 参数约 `0.393M`
- 占比约 `0.61%`

这让我第一次非常直观地感觉到：

**LoRA 真的只是在训练一个很小的补丁。**

### 这一步的核心结论

LoRA 适合：

- 人设
- 风格
- 小范围领域适配

LoRA 不适合：

- 从零补大量知识
- 重建一个太弱的底座

换句话说：

LoRA 更像“修风格”，不是“重造大脑”。

## 第 17 部分：我对 LoRA 数据集设计的理解

做完第一版 LoRA 后，我后来意识到：

如果想让 LoRA 效果更明显，数据集不能只写“你是谁？”这种身份问答。

更好的 LoRA 数据集应该包含 4 类样本：

1. 身份锚点  
2. 风格样本  
3. 普通任务样本  
4. 边界样本  

### 为什么要这样设计

因为如果数据集里几乎全是：

- 你是谁？
- 你叫什么名字？

模型很容易变成“见什么都先自我介绍”。

但如果数据里同时有：

- 普通解释任务
- 推荐任务
- 不确定时如何回答

它学到的就不是“死背身份”，而是：

**在各种问题里都带着同样的风格和边界去回答。**

这也是后来我理解 LoRA 很重要的一点：

LoRA 学到的是“输出偏移模式”，所以数据集必须覆盖这种偏移真正想发生的场景。

## 第 18 部分：我一路走下来，真正建立起来的主线认知

这部分是最重要的总结。

### 1. LLM 最核心的训练目标是什么

**预测下一个 token。**

### 2. `Pretrain` 解决什么问题

让模型学语言、知识、续写能力。

### 3. `SFT` 解决什么问题

让模型学助手问答行为和对话格式。

### 4. `LoRA` 解决什么问题

在保留底座的前提下，低成本地做风格/身份/垂域微调。

### 5. 为什么 `pretrain` 强弱会影响 `sft`

因为：

- `pretrain` 决定底座
- `sft` 决定行为

如果底座弱，行为很容易变成“更像助手地胡说”。

### 6. 为什么 `sft` 只训练 assistant 很重要

因为聊天模型的目标是：

“给上下文，生成 assistant 回复”

不是：

“背整段对话”

### 7. 为什么不能一开始就跳 LoRA / RL / Agent

因为那样变量太多，很容易看不清：

- 现在的能力来自哪一层
- 是底座问题，还是后训练问题

## 第 20 部分：这次学习里，我以后最该反复回看的 10 个结论

1. 先确认主线，再动手，不要一上来乱跑命令。  
2. `Pretrain` 是学语言和知识底座。  
3. `SFT` 是学助手问答行为。  
4. `LoRA` 是轻量补丁，不是重建底座。  
5. 大语言模型最核心的训练目标是 `next token prediction`。  
6. `loss` 本质上是“预测下一个 token 错得有多严重”。  
7. `SFT` 的关键不在于公式变了，而在于“哪些 token 参与 loss”变了。  
8. `.pth` 权重不是文本，不能拿来直接“打开看内容”，而是要被程序加载。  
9. 同一个模型结构，换不同训练阶段的权重，行为会明显不同。  
10. 训练项目里很多卡点不是模型原理，而是工程问题：路径、环境、显卡架构、数据目录、断点续训。
