这段时间做项目的过程，比单纯学概念更能暴露自己的真实水平。因为一旦进入项目环境，问题就不再只是“会不会写某一段代码”，而是会同时遇到部署、网络、依赖、代理、框架理解、结构设计、交付表达这些问题。回看这一阶段，我的项目学习大致可以分成三条主线：个人博客与 Web 项目、智能体与大模型相关项目、以及蜜罐与安全方向的综合型项目。

## 一、从个人博客入手，开始真正理解“项目不是代码堆起来就行”

博客项目是我较早开始持续投入的项目之一。最开始我主要是在理解前后端分离的基本结构，比如前端页面如何发请求、后端如何通过路由接收请求、业务逻辑如何下沉到服务层、数据库又如何配合工作。通过 FastAPI、Vue、SQLite 这类技术组合，我对一个基础 Web 项目的完整链路有了更清晰的认知。

后来在继续优化博客时，我又接触到了部署层面的问题。比如 Vercel 在国内访问不稳定，于是我尝试迁移到 GitHub Pages。这件事本身不算特别复杂，但它让我第一次比较明确地认识到：项目的好坏不只是看“本地能不能跑”，还要看部署后的可达性、稳定性和使用体验。也正是在这个过程中，我对 Git、GitHub、工作流、静态站点部署这些内容慢慢熟悉起来。

## 二、项目能力的提升，往往来自“把一个东西跑起来”

如果只看教程，很多技术都容易停留在“我知道它是什么”；但真正把项目部署起来之后，理解会完全不同。比如 hello-agents、智能旅行助手、深度研究项目、local-rag、minimind、Qwen 相关项目，这些内容虽然方向不完全一样，但给我的共同训练是：先搭环境、再跑通流程、最后再去理解内部逻辑。

在部署 minimind 和 Qwen2.5-3B-Instruct 的过程中，我对大模型项目的认识明显比以前更具体了。以前我知道 Transformer、token、微调这些词，但它们更多是概念；真正跑过项目之后，我才开始理解从文本到 token、从 embedding 到多层 Transformer、再到输出下一个 token 的完整过程，也开始知道 LoRA、QLoRA 这类微调方式到底在解决什么问题。

同样，RAG 相关项目也让我有了更实际的理解。以前容易把 RAG 想成“让模型变聪明”，后来逐渐意识到，它更像是在给模型增加可控的外部知识来源，让回答更贴近资料、减少胡说。这个认识变化很重要，因为它影响了我后面对 AI 项目的定位：不是盲目追求“模型无所不能”，而是思考怎样把模型接进一个有边界、有流程的系统里。

## 三、蜜罐项目是这一阶段最综合的一次锻炼

如果说博客项目更多训练的是 Web 基础，模型项目更多训练的是 AI 工程意识，那么蜜罐项目就是把安全、项目管理、部署、表达和交付几件事揉在一起的一次综合训练。

在这个项目里，我接触到的内容包括：主控节点和被控节点的部署、Docker 环境、内网穿透、远程服务器协同、高交互蜜罐、智能体蜜罐、攻击链分析、实验数据展示、论文与项目设计书撰写、答辩 PPT 制作等。它不是单纯地“把程序跑起来”，而是需要一边做系统，一边思考如何讲清楚创新点和实验结果。

这个过程里有不少典型问题。比如节点部署不是命令写错，就是代理和网络环境出问题；项目文档从 Markdown 合并到 Docx 时会出现公式和格式错乱；截图、页面展示、图片脱敏、作品中学校信息隐藏这些细节，也都直接影响最后交付质量。正是这些问题让我慢慢理解，项目能力其实很大一部分体现在“收尾能力”上，也就是能不能把一个东西从可运行，推进到可展示、可复现、可交付。

## 四、项目学习中最值得记住的几条经验

第一，环境问题永远不能低估。  
很多时候项目跑不起来，并不是核心代码真的有大问题，而是代理、端口、路径、权限、防火墙、容器网络这些基础条件没理顺。

第二，先跑通，再优化。  
很多新项目如果一上来就追求理解全部细节，效率会很低。更合理的方式往往是先把项目部署成功，形成整体感，再回来拆模块。

第三，项目结构决定后续维护难度。  
无论是博客项目还是蜜罐项目，随着功能变多，分层、工作流、启动方式、文档组织都会越来越重要。

第四，文档和表达能力本身就是项目能力。  
设计书、论文、PPT、实验描述、创新点总结，这些并不是附属品，而是项目最终能否被别人理解和认可的重要部分。

## 五、这段项目学习带来的成长

回看这段时间，我觉得自己最大的变化有三点。

第一，不再只把项目看成“写代码”。现在会更自然地把部署、调试、展示、复盘也视作项目的一部分。  
第二，面对新项目时没有以前那么慌了。虽然还是会卡，但至少知道可以先看依赖、看启动方式、看目录结构、看网络和环境。  
第三，对 AI 项目和安全项目之间的结合开始有了更明确的兴趣方向，比如智能体、安全助手、RAG、自动化分析、蜜罐联动这些方向。

## 六、下一步更值得补强的地方

后续如果想把项目能力继续往上提，我觉得重点应该放在以下几方面：

1. 把已经做过的几个项目整理成更标准的复盘文档，而不只是零散记录。
2. 加强对 Docker、WSL、Linux、网络代理、端口转发这些工程基础的掌握。
3. 对 FastAPI、前端框架、API 设计、数据库和部署流程再做一次系统梳理。
4. 在 AI 项目里进一步加强对 RAG、Agent、微调、评测和实际落地场景的理解。

整体来看，这段项目学习最大的意义，不只是做成了几个具体项目，而是让我逐渐建立起了一种工程视角：一个项目真正有价值，不只是能运行，还要能解释、能部署、能复现、能交付。
