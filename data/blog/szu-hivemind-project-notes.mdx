# SZU-HiveMind 项目笔记：先把这个项目整体看懂

这份笔记不是正式项目书，而是我后面自己复习、交接、答辩准备时都能直接翻的版本。  
写法尽量按“先讲人话，再讲结构，再讲技术，最后讲坑点”的顺序来，不追求论文味，追求一眼能看懂。

---

## 1. 先用一句人话说清楚：这个项目是什么

`SZU-HiveMind` 本质上是一个**面向校园网络安全运营场景的综合蜜罐平台**。

它不是那种只会假装一个协议端口、只记几条日志的小蜜罐，而是想把下面这些东西串成一套完整链路：

- 传统协议蜜罐
- 高交互容器蜜罐
- 智能体蜜罐
- 分布式主控/被控节点协同
- 攻击审计、告警通知、GeoIP、威胁情报、备份恢复

所以更准确地说，它不是“一个协议模拟器”，而是一个**能长期运行、能多节点管理、能做安全运营闭环的蜜罐系统**。

---

## 2. 这个项目到底在做什么

如果只看表面，这个项目像是在“部署很多诱饵服务”；  
如果看深一点，它真正做的是下面这件事：

**把真实攻击者引进来，记录他们的行为，再把这些行为整理成可分析、可告警、可复盘的安全数据。**

这个目标又可以拆成 4 层理解：

### 2.1 第一层：把攻击者吸引进来

系统支持多种蜜罐形态：

- 传统协议蜜罐：如 `SSH / Telnet / HTTP / FTP / MySQL / Redis / MongoDB`
- 高交互蜜罐：通过容器提供更真实的交互环境
- 智能体蜜罐：把外部智能体服务接进来，做更动态的交互诱捕

也就是说，项目不是只靠“假 banner”糊弄人，而是试着在不同深度上都给攻击者一些可交互目标。

### 2.2 第二层：把攻击过程记下来

攻击者一旦访问系统，平台会尽量把这些信息沉淀下来：

- 会话
- 攻击日志
- 告警
- 来源 IP 地理位置
- 关联情报结果
- 系统审计记录

这样后面就不只是“知道有人打过来”，而是能进一步看：

- 他打了哪些端口
- 试了什么口令
- 发了哪些请求
- 有没有明显的扫描、暴破、投递 payload 行为

### 2.3 第三层：把多节点数据聚起来

这个项目一个比较关键的点是，它不是默认只跑在一台机器上。

它支持：

- 主控节点：统一查看、统一控制、统一聚合
- 被控节点：各自运行蜜罐和本地服务

所以你可以把它理解成：

- 被控节点负责“前线接敌”
- 主控节点负责“后方看全局”

这样做的好处很明显：一台机器看到的是局部，多个节点聚合后才能看出更完整的攻击面和攻击链路。

### 2.4 第四层：把攻击数据变成运营能力

这个项目最后不是停在“抓日志”这一步，而是继续往后做：

- 告警规则
- 通知通道
- 定时报告
- 备份恢复
- 威胁情报查询
- GeoIP 地理解析
- 攻击链分析

所以它的目标已经接近一个小型安全运营平台，而不是单点实验脚本。

## 3. 项目整体架构与模块分工

看这个项目，最容易乱的地方是：文件多、模块多、节点也多。  
所以先把它拆成几个角色来看。

### 3.1 前端控制台

前端在 `SZU-HiveMind-FrontEnd/`，主要负责：

- 登录和页面展示
- 蜜罐管理
- 节点管理
- 威胁审计
- 告警通知
- GeoIP、备份、系统管理

技术上它是一个 React + TypeScript 的前端，但从职责上说，它就是整个平台的“操作台”。

### 3.2 后端服务

后端在 `SZU-HiveMind-BackEnd/`，主要负责：

- 提供 REST API
- 管理用户、权限、配置
- 控制蜜罐实例生命周期
- 写入会话、日志、告警、审计
- 提供分布式节点聚合和转发能力

如果前端是操作台，后端就是整套平台的“大脑和调度中心”。

### 3.3 蜜罐运行层

这层是真正接流量的地方，包括：

- 传统协议蜜罐
- Docker 高交互蜜罐
- 智能体蜜罐

它们不一定都运行在同一台机器上。  
有的更适合本地跑，有的更适合 Docker worker，有的更适合放到云上暴露出去。

### 3.4 数据与模板层

项目把很多关键运行数据都收口到根目录 `data/` 下：

- 主数据库
- 模板数据库
- 指纹模板
- GeoIP 数据
- 备份数据

这一层很重要，因为它关系到：

- 备份能不能做
- 迁移会不会丢数据
- 同步代码时会不会误覆盖现场数据

### 3.5 文档与脚本层

这个项目还有一层经常被忽略，但实际上特别有用：

- `README.md`：总览
- 前后端 README：交接说明
- `docs/快速自查命令.md`：日常排障速查
- `docs/创建启动命令.md`：更完整的运维说明
- `scripts/`：截图、导出、图表、演示数据、文档处理脚本

所以它不仅有“系统本体”，还有一套比较完整的维护和文档产出链路。

## 4. 运行链路与部署形态

这一部分很重要，因为这个项目不是“随便开两个服务就完事”的那种。

### 4.1 默认入口是 `start.sh`

这个项目的统一运维入口是仓库根目录的：

```bash
cd /home/tlx/SZU-HIVEMIND
./start.sh setup
./start.sh start
./start.sh status
./start.sh restart
./start.sh logs
./start.sh stop
```

平时最该记住的是三条：

- `./start.sh start`
- `./start.sh status`
- `./start.sh restart`

也就是说，默认思路不是分别手搓前端、后端、节点，而是优先让 `start.sh` 统筹。

### 4.2 前后端端口怎么配合

默认端口关系如下：

- 前端对外入口：`54321`
- 后端 API：`8000`

访问路径大致可以理解成：

1. 浏览器访问前端 `http://<host>:54321/`
2. 前端请求 `/api/v1/...`
3. `server.js` 再把 `/api/*` 代理到 `http://127.0.0.1:8000`

所以很多“页面能开，但接口炸了”的问题，本质上都落在这条链路上。

### 4.3 单机模式和分布式模式

这个项目同时支持两种理解方式：

- 单机模式：前端、后端、蜜罐都在当前机器上
- 分布式模式：主控节点统一接多个 worker

分布式时最核心的角色分工是：

- **主控节点**：聚合数据、远程控制、统一展示
- **被控节点**：本地跑服务、本地写数据、接受主控管理

不要把它们理解成“谁都一样，只是换台机器跑”，  
因为主控和被控在职责上本来就不对等。

### 4.4 当前项目为什么比较像“工程原型”

因为它已经不是只停在开发机实验，而是考虑了下面这些现实问题：

- Docker worker 恢复
- 云上 worker 自动启动
- 集群状态检查
- 数据备份恢复
- 低配部署下前端预构建分发
- 截图与文档导出

也就是说，这套项目已经明显带有“要交付、要演示、要维护”的工程味道。

## 5. 主要技术栈与各自作用

这部分不要只背名字，关键是知道“它们分别在这个项目里干什么”。

### 5.1 Django + DRF

后端核心是 `Django + Django REST Framework`。

它们在这里主要负责：

- 用户认证
- 管理后台
- REST API
- 数据模型和数据库操作
- 各类系统管理接口

如果把项目比作一套安全平台，Django 就是后端骨架。

### 5.2 React + TypeScript

前端核心是 `React + TypeScript`。

作用主要是：

- 做控制台界面
- 管理页面状态
- 对接后端接口
- 提供节点、蜜罐、日志、告警等页面

`TypeScript` 的意义在这里不是“更高级”，而是让页面逻辑和接口数据更不容易写乱。

### 5.3 Vite + Node `server.js`

这个项目前端既有开发模式，也有生产模式：

- 开发时常用 `Vite`
- 生产时用 `server.js` 提供静态文件并反代后端 API

所以你可以理解成：

- `Vite` 负责开发体验
- `server.js` 负责最终对外访问

### 5.4 SQLite / 模板数据库 / `data/`

这个项目的数据管理有一个很实际的特点：  
很多关键运行数据都统一收口到了 `data/`。

这里面包括：

- 主数据库
- 模板数据库
- GeoIP 数据
- 备份
- 配置文件

这说明项目很强调“方便迁移、方便备份、方便交接”。

### 5.5 Docker

Docker 在这个项目里不是装饰品，而是高交互蜜罐的重要支撑。

它主要用于：

- 高交互蜜罐镜像管理
- 高交互实例运行
- Docker worker 节点承载

所以如果高交互那条链路要稳定，Docker 这一层必须健康。

### 5.6 智能体接入与 DeepSeek 适配器

这个项目比较特别的一点，是它不只做传统协议蜜罐，还做智能体蜜罐。

这里涉及几部分能力：

- 智能体端点管理
- `health` / `simulate` 标准化接入
- Mock 智能体接口
- 外部大模型适配器

可以把这条链路理解成：

- 平台先定义统一接入方式
- 再通过适配器把平台请求转换成大模型服务能理解的格式

这样后面要换不同模型，不用把整个平台推倒重来。

### 5.7 Playwright / Pandoc / WPS 文档链路

这个项目连“文档怎么产出”都想到了。

常见工具包括：

- `Playwright`：自动截图 README 功能页
- `pandoc`：Markdown 转 Docx
- `postprocess_docx_for_wps.py`：修正 WPS 下图表和段落格式

这说明项目不仅重视系统功能，也重视最终展示材料怎么稳定产出。

## 6. 这个项目已经做出来的创新点

这部分我不按比赛稿那种很正式的口气写，而是按“我自己记住项目亮点”的方式说。

### 6.1 创新点一：把智能体接进蜜罐体系

传统蜜罐很多时候都是固定模板、固定脚本，交互比较死。  
这个项目做的第一件有意思的事，是把外部智能体服务抽象成标准化端点接进来。

它的关键价值在于：

- 不再只靠静态回复
- 可以做更动态的交互诱捕
- 可以统一管理端点、探测健康状态、编排实例

简单说，就是把“死板协议模拟”往“可扩展智能交互诱捕”推进了一步。

### 6.2 创新点二：开始盯 AI 基础设施攻击面

这个项目不是只盯传统 SSH、HTTP、数据库暴露面，  
它还往前看了一步，开始考虑：

- OpenAI 兼容接口
- Ollama 风格接口
- AI 推理 API
- 提示词注入、工具滥用、模型枚举这类新风险

这个方向的价值在于：

- 更贴近现在的新型暴露面
- 能把 AI 相关访问流量从“普通 HTTP 请求”里单独识别出来
- 为以后做 AI 安全诱捕预留空间

### 6.3 创新点三：从单条日志走向攻击链分析

很多系统只能告诉你“有一条攻击日志”。  
这个项目更想做的是：

- 把多个节点上的攻击会话串起来
- 做跨节点、跨阶段关联
- 提取 IOC、TTP 和风险摘要

也就是说，它的方向已经不是“只收集样本”，而是向“自动生成安全情报”靠近。

### 6.4 这三个创新点怎么连成一条线

我觉得这个项目最顺的理解方式是：

1. 先用传统蜜罐和高交互蜜罐做广覆盖
2. 再用智能体蜜罐提高动态交互和新暴露面诱捕能力
3. 最后把多节点数据串成攻击链，做更高层的分析输出

这样它就不是 3 个零散亮点，而是一条逐步升级的系统路线。

## 8. 注意事项与常见坑

这一节是最值得后面反复看的，因为很多问题不是“不会写代码”，而是“项目真的跑起来以后很容易踩坑”。

---

### 8.1 部署 / 运行类注意事项

#### 1. 默认优先走 `./start.sh`，不要一上来就手工分开乱拉服务

这个项目现在的统一入口就是：

```bash
cd /home/tlx/SZU-HIVEMIND
./start.sh start
./start.sh status
./start.sh restart
```

很多排障第一步也不是直接钻到某个子目录里，而是先看：

- `./start.sh status`
- `./start.sh logs`

因为它们已经把前后端、Docker worker、云端 worker 这些常见状态收口过了。

#### 2. 页面能打开但 API 不通，优先查 `8000`

这个问题特别典型：  
前端 `54321` 能访问，不代表后端 API 一定正常。

如果出现 `502` 或“页面壳子在，数据全空”，优先排查：

- 后端是不是还在 `8000` 正常运行
- `BACKEND_URL` 有没有指错
- `server.js` 的代理链路有没有断

最实用的检查思路：

```bash
./start.sh status
curl -I http://127.0.0.1:54321/
curl -I http://127.0.0.1:8000/admin/
```

#### 3. 改完配置后，经常不是“自动生效”，而是要重启

这个项目里很多关键配置改完都应该默认想到：

```bash
./start.sh restart
```

尤其是这些情况：

- 改了 `config.env`
- 改了端口
- 改了分布式配置
- 改了 Mock 智能体开关
- 改了前端构建产物

如果改完忘了重启，很容易出现“我明明改了，为什么页面还是旧的”。

#### 4. GeoIP 一旦缺文件，很多地图和解析效果会直接不对

如果发现：

- GeoIP 页面异常
- 地图不显示
- IP 地理位置解析明显不对

优先检查：

- `data/geoip/GeoLite2-City.mmdb` 是否存在
- GeoIP 数据是否完整

必要时可以重新下载或刷新相关数据。

#### 5. 低配机器不要硬编前端，容易 OOM

这个项目已经明确提醒过：  
前端更推荐在开发机构建好，再把产物分发到部署机。

原因很现实：

- 低配机器本地构建前端容易内存不够
- 一旦 OOM，构建失败不说，还会拖慢排障

所以更稳的思路是：

- 开发机 `rebuild`
- 服务器只部署构建产物

---

### 8.2 分布式维护类注意事项

#### 1. 主控和被控不是一回事，别混着理解

这个项目的分布式模式里：

- 主控负责统一查看、统一聚合、统一控制
- 被控负责本地运行、本地数据沉淀、对外接受主控接入

所以很多配置和操作不能混用。

最容易犯的错就是：

- 把 worker 的配置思路套到主控上
- 把主控的接入参数拿去给 worker 直接复用

#### 2. 节点信息改了以后，要重新保存连接配置

如果这些信息变了：

- `host`
- `port`
- `shared_key`
- `route_key`
- `api_base_path`

那主控侧通常不能假设“自动全跟上了”，  
而要重新保存或重新确认节点配置。

不然就很容易出现：

- 服务其实在线
- 但主控看起来始终离线

#### 3. 代码同步时不要误覆盖 `data/`

这是非常关键的一条。

项目里很多现场运行数据都在 `data/`：

- 主数据库
- 模板数据库
- GeoIP
- 备份

如果同步代码时把 `data/` 一起粗暴覆盖，后果往往不是“小问题”，而是直接把现场数据状态搞乱。

所以默认原则应该是：

- 同步代码可以
- 覆盖运行数据要格外慎重

#### 4. 节点离线时先看集群状态，不要上来就乱重建

日常看节点，优先还是：

```bash
./start.sh status
```

如果看到远端 worker 离线，再逐层排查：

- 主控本地前后端是否正常
- Docker worker 是否在线
- 云端 worker 是否公网可达
- 远端自启动服务是否接住

不要一看到离线就立刻乱删、乱配、乱建节点。

#### 5. 主控适合做聚合和管理，不一定适合替代所有节点本地操作

有些能力主控看起来能统一看到，但不代表所有现场调试都应该只从主控页面做。

尤其是：

- Docker worker 内部服务自检
- 云端适配器服务健康检查
- 节点本机端口监听情况

这些问题很多时候还是得回到对应节点本机确认。

---

### 8.3 文档 / 截图 / 导出类注意事项

#### 1. 截图前先确认站点真的能访问

README 截图不是手工裁的，而是通过无头浏览器自动采集。

默认入口是：

```bash
./scripts/capture_readme_assets.sh
```

执行前至少先确认：

- `http://127.0.0.1:54321` 能打开
- 管理员凭据配置正确

不然截图脚本一开始就会失败。

#### 2. Playwright 依赖缺失会让截图链路直接断掉

截图这条链路依赖 `Playwright` 和对应浏览器环境。  
如果缺少这些依赖，常见现象就是：

- 浏览器拉不起来
- 页面采集任务直接失败
- 日志里出现浏览器或系统库相关错误

所以截图失败时，不能只盯页面本身，也要检查截图运行环境。

#### 3. 页面选择器漂移时，自动截图会失效

这个问题很真实：

- 页面结构一改
- 按钮文案一变
- 某个 tab 默认行为变了

之前的截图任务就可能定位不到目标元素。

这种情况下要去看：

- `logs/capture-readme-assets-*.json`
- `scripts/capture_manifest.json`

很多时候不是系统坏了，而是截图脚本里写死的页面选择器失效了。

#### 4. Markdown 导出到 WPS 不是“直接转一下就一定完美”

这个项目的 Markdown -> Docx 导出链路依赖：

- `pandoc`
- 参考 Docx
- `postprocess_docx_for_wps.py`

也就是说，真正稳定的流程不是只跑转换，而是：

1. `pandoc` 导出
2. 后处理脚本修格式
3. 再交给 WPS 打开

如果缺这几步中的任意一环，就可能出现：

- 图表位置乱
- 段落样式不稳定
- 标题颜色和间距不对
- 表格在 WPS 里显示很怪

#### 5. 文档里不要直接留真实敏感信息

这一条虽然像常识，但真的要单独记住。

后面如果要写命令、配置示例、接口调用示例，默认都应该写成：

- ``<ADMIN_USERNAME>``
- ``<ADMIN_PASSWORD>``
- ``<TOKEN>``
- ``<API_KEY>``
- ``<SHARED_KEY>``

不要把真实密码、Token、SecretKey、云凭据直接写进长期保存的笔记里。

这既是安全问题，也是交接卫生问题。

## 9. 我自己复习这个项目时，最该先记住什么

如果后面时间很赶，只允许我先记最核心的几件事，我会先记这 6 条：

1. 这是一个分布式校园蜜罐运营平台，不只是单机蜜罐。
2. 统一入口优先走 `./start.sh`，别先手搓一堆子命令。
3. 前端端口是 `54321`，后端 API 是 `8000`。
4. 主控负责聚合和控制，被控负责本地运行和接入。
5. `data/` 是运行数据重地，不能同步代码时乱覆盖。
6. 文档截图和 WPS 导出都有专门脚本链路，不是“手点就行”。

如果把这几条先抓住，再回头看其余细节，这个项目就不会显得那么乱。
